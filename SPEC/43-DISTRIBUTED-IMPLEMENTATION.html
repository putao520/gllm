§0 动机与约束

§0.1 问题

gllm 当前仅支持单 GPU 推理。大模型（70B+）和 MoE 模型（A22B+）需要多 GPU 协同推理。gllm-nccl 已提供底层通信原语（AllReduce/AllGather/ReduceScatter/AllToAll/RDMA），但 gllm 侧缺少完整的分布式集成层：通信句柄生命周期、TP 权重分片、通信-计算重叠调度、PD 分离路由、KV 跨节点传输、MoE 分布式 dispatch、Context Parallelism、分布式推测解码。

§0.2 约束

  • 三仓约定: gllm → gllm-kernels → gllm-nccl 单向依赖。gllm 通过 gllm-kernels VmInstr call stub 间接使用 gllm-nccl，不直接依赖。
  • ARCH-RUST-IS-CODEGEN: 分布式通信必须 JIT 编译为 VmInstr call stub，禁止 Rust 运行时编排通信。
  • ARCH-UNIFIED-COMPILER: 分布式图构建走统一 CompilerGraph，编译器通过 GraphTopologyAnalysis 推导通信点，不前置 bool flag。
  • NO-SILENT-FALLBACK: 通信失败必须返回 Err，禁止静默降级到单 GPU。
  • ARCH-JIT-YIELDS: 通信策略顺从硬件拓扑（NVLink vs PCIe vs RDMA），不要求硬件适配代码。

    §1.1 REQ-DIST-001: CommHandle 生命周期管理

    REQ-DIST-001 CommHandle 生命周期管理

    描述: Executor 构建时根据 ParallelConfig 初始化 NCCL 通信句柄。CommHandle 封装 gllm-nccl CommHandle，提供 rank()/world_size()/is_distributed()/destroy() 方法。Drop trait 实现安全释放（先 barrier 再 destroy）。

    前置条件:

    • ParallelConfig 已配置（tp_size >= 1, rank 有效）
    • gllm-nccl feature 已启用（cfg(feature = "nccl")）
    • CUDA/RoCm runtime 已初始化

    • world_size() 返回通信组总大小

    验收标准:

    • 单 GPU: is_distributed() == false, rank() == 0, world_size() == 1
    • 多 GPU: is_distributed() == true, rank 值域 [0, world_size)
    • Drop 无资源泄漏（ncclCommDestroy 被调用）
    • 重复 Drop 安全（幂等）

    §2 Layer 1: Tensor Parallelism

    §2.1 REQ-DIST-004: TP 权重分片

    REQ-DIST-004 TP 权重分片加载

    描述: 按 rank 对权重张量执行分片，权重加载时执行。分片策略：

    • 列切分: QKVProj/OProj — 按 num_heads/tp_size 切分列维度，每个 rank 持有 1/tp_size 的 head
    • 行切分: GateUpProj/DownProj — GateUpProj 按行切分（每个 rank 持有 1/tp_size 行），DownProj 按列切分
    • 列切分: lm_head — 按 vocab_size/tp_size 切分列维度
    • 不分片: embedding/RMSNorm/RoPE — 所有 rank 持有完整副本

    前置条件:

    • CommHandle 已初始化（REQ-DIST-001）
    • 权重文件已加载到内存
    • ModelConfig 包含 num_heads/num_kv_heads/head_dim 信息

    后置条件:

    • 每个 rank 持有对应分片的权重子张量
    • 分片后权重 shape 正确（QKVProj: [hidden, num_heads_per_rank * head_dim]）
    • 不分片的权重在所有 rank 上完全一致

    验收标准:

    • tp_size=2 时 QKVProj 列数减半
    • tp_size=4 时 GateUpProj 行数减为 1/4
    • 所有 rank 的 embedding 权重 bit-wise 一致
    • 分片后模型 forward 数值与单 GPU 等价（在 AllReduce 聚合后）

    关联 Entity: ENT-DIST-TP-SHARD

    关联 Dataflow: DF-DIST-002

    验收标准：
    1. tp_size=2 时 QKVProj 列数减半
    2. tp_size=4 时 GateUpProj 行数减为 1/4
    3. 所有 rank 的 embedding 权重 bit-wise 一致
    4. 分片后模型 forward 数值与单 GPU 等价（在 AllReduce 聚合后）

    TP 权重分片加载：按 Tensor Parallelism 策略将模型权重切分到各 GPU rank。分片规则：(1) Attention QKVProj/OProj — 按 num_heads/tp_size 切分列维度，每个 rank 持有 1/tp_size 的 head；(2) FFN GateUpProj/DownProj — GateUpProj 按行切分（每个 rank 持有 1/tp_size 行），DownProj 按列切分；(3) lm_head — 按 vocab_size/tp_size 切分列维度；(4) embedding/RMSNorm/RoPE — 所有 rank 持有完整副本（replicated）。分片在 weight_loader 加载时按 rank 过滤，无需运行时通信。

    tp_size=2 时 QKVProj 列数减半
    tp_size=4 时 GateUpProj 行数减为 1/4
    所有 rank 的 embedding 权重 bit-wise 一致
    分片后模型 forward 数值与单 GPU 等价（在 AllReduce 聚合后）

    §2.2 REQ-DIST-005: TP AllReduce 通信

    REQ-DIST-005 TP AllReduce 通信

    描述: 每个 Transformer 层 forward 后调 comm_handle.all_reduce(Sum) 聚合部分结果。行切分层（GateUpProj→DownProj）的输出需要 AllReduce 聚合；列切分层（QKVProj→OProj）的输出在 OProj 后需要 AllReduce。通信点由 CompilerGraph 的 GraphTopologyAnalysis 自动推导，不前置 bool flag。

    前置条件:

    • TP 权重分片完成（REQ-DIST-004）
    • CommHandle 可用
    • CompilerGraph 包含 AllReduce VmInstr call stub

    后置条件:

    • AllReduce(Sum) 后每个 rank 持有完整的聚合结果
    • 聚合结果与单 GPU forward 数值等价（在浮点舍入误差内）
    • 通信延迟不阻塞后续独立计算

    验收标准:

    • 2-GPU TP: AllReduce 后 logits 与单 GPU 差异 < 1e-5
    • 4-GPU TP: AllReduce 后 logits 与单 GPU 差异 < 1e-4
    • 通信点数量 = num_layers * 2（attention + FFN 各一次）

    关联 Entity: ENT-DIST-TP-COMM

    关联 Dataflow: DF-DIST-002

    关联 Controlflow: CF-DIST-001

    验收标准：
    1. AllReduce 后各 rank hidden_state 数值一致
    2. AllReduce 通信延迟与 payload 大小线性相关
    3. 单 GPU 模式下无 AllReduce 通信开销

    TP AllReduce 通信：Attention 计算后各 rank 持有部分 head 的输出，需 AllReduce 求和还原完整 hidden_state。CompilerGraph 包含 AllReduce VmInstr call stub，JIT 编译时生成 NCCL AllReduce 调用点。通信发生在每个 Transformer 层的 Attention 输出后。

    AllReduce 后各 rank hidden_state 数值一致
    AllReduce 通信延迟与 payload 大小线性相关
    单 GPU 模式下无 AllReduce 通信开销

    §2.3 REQ-DIST-006: TP 列并行通信

    REQ-DIST-006 TP 列并行通信

    描述: ReduceScatter+AllGather 替代 AllReduce 用于列切分层，支持通信-计算重叠。列并行模式：QKVProj 列切分后各 rank 独立计算 attention → ReduceScatter 聚合 → 本地 OProj → AllGather 广播。相比 AllReduce 减少通信量 50%（每个 rank 只发送 1/tp_size 数据）。

    前置条件:

    • TP 权重分片完成（REQ-DIST-004）
    • CommHandle 支持 ReduceScatter/AllGather
    • 通信-计算重叠调度器可用（REQ-DIST-007/008）

    后置条件:

    • ReduceScatter 后每个 rank 持有 1/tp_size 的聚合结果
    • AllGather 后每个 rank 持有完整结果
    • 通信量相比 AllReduce 减半

    验收标准:

    • 列并行数值结果与 AllReduce 等价
    • 通信量 = 2 * batch_size * hidden_size / tp_size（vs AllReduce = 2 * batch_size * hidden_size）
    • 与重叠调度器配合时通信延迟部分隐藏

    关联 Entity: ENT-DIST-TP-COMM

    关联 Dataflow: DF-DIST-003

    关联 Controlflow: CF-DIST-001

    验收标准：
    1. ReduceScatter 后各 rank 持有对应分片的 FFN 输出
    2. AllGather 后各 rank 持有完整 FFN 输出
    3. ReduceScatter+AllGather 组合与单次 AllReduce 数值等价

    TP 列并行通信：FFN 层采用列并行策略，GateUpProj 按行切分后各 rank 独立计算，输出需 ReduceScatter 汇聚到对应 rank；DownProj 按列切分后各 rank 计算部分结果，需 AllGather 还原完整输出。ReduceScatter + AllGather 组合实现列并行 FFN 的完整通信。

    ReduceScatter 后各 rank 持有对应分片的 FFN 输出
    AllGather 后各 rank 持有完整 FFN 输出
    ReduceScatter+AllGather 组合与单次 AllReduce 数值等价

    §3 Layer 2: 通信-计算重叠

    §3.1 REQ-DIST-007: 双缓冲重叠调度

    REQ-DIST-007 双缓冲通信-计算重叠

    描述: ForceDoubleBuffer mk_variant 返回 DoubleBufferCluster，交替调度通信/计算 SM 分区。双缓冲策略：Layer N 的通信与 Layer N+1 的计算重叠执行。通信使用专用 SM 分区，计算使用剩余 SM 分区。两个 buffer 交替使用，消除通信等待时间。

    前置条件:

    • TP 通信已实现（REQ-DIST-005/006）
    • GPU 支持 SM 分区（ComputeCapability >= 7.0）
    • Mega-Kernel 支持多 stream 调度

    后置条件:

    • DoubleBufferCluster 包含两个交替 buffer
    • 通信和计算在时间轴上重叠
    • 每个 step 的有效通信延迟趋近于 0（完全隐藏时）

    验收标准:

    • 双缓冲模式下 step 延迟 < 串行模式延迟 * (1 + 通信/计算重叠率)
    • 重叠率 >= 70% 时有效通信延迟 < 30% 原始通信延迟
    • 数值结果与串行模式完全一致

    关联 Entity: ENT-DIST-DOUBLE-BUFFER

    关联 Dataflow: DF-DIST-003

    关联 Controlflow: CF-DIST-002

    验收标准：
    1. 双缓冲模式下通信延迟完全隐藏在计算时间内
    2. 无双缓冲时延迟 = compute + comm，有双缓冲时延迟 ≈ max(compute, comm)
    3. SM 分区正确隔离通信和计算 stream

    双缓冲通信-计算重叠：通信 stream 与计算 stream 并行执行。当 rank N 在计算当前 chunk 的 FFN 时，通信 stream 同时传输上一个 chunk 的 AllReduce 结果。双缓冲策略将通信延迟隐藏在计算时间内，理论加速比 = compute_time / max(compute_time, comm_time)。

    双缓冲模式下通信延迟完全隐藏在计算时间内
    无双缓冲时延迟 = compute + comm，有双缓冲时延迟 ≈ max(compute, comm)
    SM 分区正确隔离通信和计算 stream

    §3.2 REQ-DIST-008: FLUX 分解调度

    REQ-DIST-008 FLUX 分解调度

    描述: ForceFlux mk_variant 返回 FluxDecompose，将 AllReduce 分解为 ReduceScatter + LocalCompute + AllGather 三阶段，实现更细粒度的通信-计算重叠。FLUX 策略连接 gllm-nccl flux_decompose()：ReduceScatter 完成部分数据后立即启动本地计算，不等全部数据到齐。AllGather 在本地计算完成后立即启动，不等所有 rank 完成。

    前置条件:

    • 列并行通信已实现（REQ-DIST-006）
    • gllm-nccl flux_decompose() 可用
    • GPU 支持 concurrent kernel execution

    后置条件:

    • FluxDecompose 将 AllReduce 分解为三阶段流水线
    • ReduceScatter→LocalCompute→AllGather 流水线化
    • 通信延迟进一步降低（相比双缓冲）

    验收标准:

    • FLUX 模式 step 延迟 < 双缓冲模式 step 延迟
    • 数值结果与串行 AllReduce 完全一致
    • ReduceScatter 完成部分数据后本地计算立即启动

    关联 Entity: ENT-DIST-FLUX

    关联 Dataflow: DF-DIST-003

    关联 Controlflow: CF-DIST-002

    1. FLUX 3 阶段流水线正确执行：ReduceScatter → LocalCompute → AllGather

    3. FLUX 分解延迟 ≈ ReduceScatter_latency + max(compute_latency, AllGather_latency)

    FLUX 分解调度：将列并行 FFN 的 ReduceScatter→LocalCompute→AllGather 三阶段流水线化。FLUX 将大粒度 AllReduce 分解为细粒度 ReduceScatter + AllGather，配合 chunk 切分实现通信-计算流水线重叠。每个 chunk 独立完成三阶段，chunk 间形成流水线。

    §3.3 REQ-DIST-009: 量化通信

    REQ-DIST-009 量化通信

    描述: CommCompressHint.AlwaysCompress/ForceQuant 驱动 QuantizedComm 策略。发送前量化（pre_send_quantize）：FP32/FP16 → INT8/FP8，减少通信量 2-4x。接收后反量化（post_recv_dequantize）：INT8/FP8 → FP32/FP16，恢复计算精度。量化格式由 DeviceProfile 和 CommCompressHint 共同决定。

    前置条件:

    • TP 通信已实现（REQ-DIST-005/006）
    • 量化算子已注册到 ScalarOpRegistry
    • DeviceProfile 支持目标量化格式

    后置条件:

    • AlwaysCompress: 所有通信自动量化/反量化
    • ForceQuant: 强制使用指定量化格式（如 FP8）
    • 通信量减少 2x（FP16→INT8）或 4x（FP32→INT8）

    验收标准:

    • 量化通信后数值误差 < 1e-3（INT8）或 < 1e-2（FP8）

    REQ-DIST-018 PP 配置与 stage 划分

    Pipeline Parallel (PP) 基础配置与 stage 划分。ParallelConfig 扩展 pp_size 字段，PipelineConfig 封装 pp_size/stage_id/num_virtual_stages/micro_batch_size。各 stage 持有连续层范围，层范围按 stage_id 均分。

    验收标准：
    1. ParallelConfig 包含 pp_size >= 1 和 stage_id 字段
    2. pp_size == 1 时退化为单设备执行（零 PP 开销）
    3. pp_size > 1 时各 stage 持有有效的 stage_id ∈ [0, pp_size)
    4. 各 stage 的层范围 [stage_id * L/pp, (stage_id+1) * L/pp) 无重叠且全覆盖
    5. PipelineConfig 可从 DistributedConfig 正确推导

    REQ-DIST-019 PP 权重分片加载

    Pipeline Parallel 权重分片：WeightLoader 根据 PipelineConfig.stage_id 计算本 stage 负责层范围，仅加载对应层权重到内存。Embedding/LayerNorm 等共享层被所有 stage 加载。

    验收标准：
    1. WeightLoader 根据 stage_id 计算层范围并仅加载该范围权重
    2. 未分配到本 stage 的层权重不占用内存
    3. 权重分片后各 stage 总内存与完整模型成 1/pp 关系
    4. Embedding/LayerNorm 等共享层被所有 stage 加载（不切分）
    5. MoE 共享专家权重随所属 stage 加载

    REQ-DIST-020 微批次调度器 MicroBatchScheduler

    微批次调度器 MicroBatchScheduler：将完整输入 batch 按 micro_batch_size 切分为多个微批次，用于流水线填充。支持动态微批次大小（最后一个微批次可能更小）。

    验收标准：
    1. MicroBatchScheduler::split(batch, micro_batch_size) 返回 Vec<MicroBatch>
    2. 微批次数量 = ceil(batch_size / micro_batch_size)
    3. 最后一个微批次可能小于 micro_batch_size
    4. micro_batch_size == 0 时返回 Err
    5. 微批次聚合结果与原始 batch 数值等价

    REQ-DIST-021 Stage 间激活/梯度传输

    Stage 间激活/梯度传输机制：相邻 pipeline stage 通过 CommHandle 的 P2P send/recv 传输中间激活和梯度。Forward 方向从前到后，Backward 方向从后到前。

    验收标准：
    1. Stage 间通过 CommHandle.send/recv 传输激活张量
    2. Forward: Stage i → Stage i+1 发送激活
    3. Backward: Stage i+1 → Stage i 发送梯度
    4. 首 stage 激活从用户输入获取，末 stage 激活为模型输出
    5. 传输延迟与激活大小线性相关
    6. 量化传输可选应用

    REQ-DIST-022 GPipe 与 1F1B 调度策略

    GPipe 与 1F1B 微批次调度策略：GPipe 先执行所有 forward 再执行所有 backward（简单但内存峰值高）；1F1B 在 warmup 后交替执行 forward/backward（内存更优）。复用 gllm-nccl PipelineParallel.strategy 的 MicroBatchStrategy 枚举。

    验收标准：
    1. GPipe 策略实现：所有 forward 后所有 backward
    2. 1F1B 策略实现：warmup forward + interleaved fwd/bwd + cooldown backward
    3. 策略切换通过 MicroBatchStrategy 枚举共享
    4. GPipe 峰值激活内存 = num_microbatches * activation_bytes
    5. 1F1B 峰值激活内存 = (pp_size - stage_id) * activation_bytes
    6. 两种策略计算结果数值等价

    REQ-DIST-023 交错 PP 调度 (Interleaved 1F1B)

    交错 Pipeline Parallel 调度（Interleaved 1F1B）：每个 device 持有多个不连续层块（virtual stages），微批次在 virtual stage 间交替发射，减少流水线气泡比例。num_virtual_stages 控制交错粒度。

    验收标准：
    1. 交错调度器按 virtual stage 顺序交替发射微批次
    2. num_virtual_stages >= pp_size，virtual stage 与 device 映射关系正确
    3. 交错模式减少 bubble ratio 至 (pp_size-1)/(num_virtual_stages*pp_size)
    4. 每个 device 持有 num_virtual_stages/pp_size 个不连续层块
    5. 交错激活内存 = 非交错激活内存
    6. 交错调度输出数值与非交错等价

    REQ-DIST-024 Pipeline Bubble 分析

    Pipeline bubble 分析与可视化指标：量化流水线气泡比例，支持非交错和交错模式的气泡率计算。提供 BubbleAnalyzer 用于调度策略决策依据。

    验收标准：
    1. PipelineBubble 计算：bubble_ratio = (pp_size - 1) / (num_microbatches * pp_size)
    2. 交错模式 bubble_ratio = (pp_size - 1) / (num_virtual_stages * num_microbatches * pp_size)
    3. 分析模型支持 pp_size ∈ [2, 64], num_microbatches ∈ [1, 1024]
    4. 分析结果用于调度器策略选择

    REQ-DIST-025 动态微批次大小调整

    动态微批次大小调整：根据运行时测量的 stage 间通信延迟和可用带宽，自适应调整 micro_batch_size。通信受限时缩小微批次以增加 overlap，计算受限时增大微批次以减少通信次数。

    验收标准：
    1. 根据 device 间延迟和带宽自适应调整 micro_batch_size
    2. 延迟高/带宽低 → 减小 micro_batch_size 以增加 overlap
    3. 延迟低/带宽高 → 增大 micro_batch_size 以减少通信次数
    4. 自适应结果不小于用户指定的 micro_batch_size 最小值
    5. 自适应调整在 prefill 开始前完成

    REQ-DIST-026 PP 通信-计算重叠

    PP 通信-计算重叠：利用 GPU 双流（计算流 + 通信流）重叠 stage 间激活传输与本地微批次计算。复用 Layer 2 的双缓冲重叠机制（REQ-DIST-007）和量化通信（REQ-DIST-009）。

    验收标准：
    1. Stage 间激活传输与本地计算通过双 stream 重叠
    2. 计算流和通信流独立并行
    3. 无重叠时延迟 = compute + comm，有重叠时延迟 ≈ max(compute, comm)
    4. 重叠模式下通信延迟隐藏率 > 80%
    5. SM 分区隔离通信和计算流
    6. 量化激活传输可选应用

    REQ-DIST-027 PP KV Cache 管理

    Pipeline Parallel 下 KV Cache 管理：各 stage 独立管理本 stage 层的 KV cache。KV page 分配按 stage_id 隔离。PP 模式下无需跨 stage KV 传输（不同于 Layer 3 的 PD 分离模式）。

    验收标准：
    1. 每个 stage 独立管理本 stage 层的 KV cache
    2. KV page 分配按 stage_id 隔离
    3. 跨 stage KV 不传输
    4. Prefix caching 在各 stage 独立工作
    5. KV cache 总内存 = 全模型 KV / pp_size

    REQ-DIST-028 PP 梯度回传与权重同步

    Pipeline Parallel 梯度回传与权重同步：backward pass 完成后各 stage 通过 P2P 通信将梯度传回前序 stage。各 stage 在收到梯度后更新权重。支持 2D (TP+PP) 并行时先做 TP AllReduce 再做 PP 梯度传递。

    验收标准：
    1. 各 stage 梯度在 backward 后通过 send/recv 传回前序 stage
    2. 权重更新在所有 stage 完成 backward 后同步执行
    3. 梯度聚合后各 stage 权重与单设备训练等价
    4. 支持与 TP 梯度聚合共存

    REQ-DIST-029 2D 并行 (TP+PP) 拓扑

    2D 并行 (TP+PP) 拓扑与通信编排：world_size = tp_size * pp_size，rank 按 (tp_rank, pp_rank) 二维映射。同一 PP stage 内的 TP 组做 AllReduce，跨 PP stage 的 P2P 通信在 TP 主 rank（tp_rank=0）间进行。

    验收标准：
    1. 支持 TP+PP 2D 并行
    2. world_size = tp_size * pp_size，rank 映射正确
    3. 每个 rank 属于一个 TP 组和一个 PP 组
    4. TP 组内 AllReduce 通信仅限同 stage
    5. PP 组内 P2P 通信跨 stage
    6. 2D 并行结果与单 GPU 等价

    REQ-DIST-030 3D 并行 (TP+PP+EP) 拓扑

    3D 并行 (TP+PP+EP) 拓扑与通信编排：在 2D 基础上增加 Expert Parallel (EP) 维度用于 MoE 分布式。EP 组内执行 AllToAll dispatch/combine，TP/PP 通信在 EP 维度上独立。

    验收标准：
    1. 支持 TP+PP+EP 3D 并行
    2. world_size = tp_size * pp_size * ep_size
    3. 3D rank 映射正确
    4. EP 组内做 MoE ExpertParallel 通信
    5. TP+PP 通信在 EP 维度上独立
    6. 3D 并行 MoE 结果与单 GPU 等价

    REQ-DIST-031 PP 与 PD 分离协同

    PP 与 PD 分离协同：Pipeline Parallel 与 Prefill-Decode 分离协同工作。Prefill 阶段利用完整 PP 流水线高效处理长序列，完成后 KV cache 按页粒度迁移到 dedicated decode stage 进行解码。

    验收标准：
    1. PP 与 PD 分离协同
    2. KV 迁移从 prefill 末 stage 到 decode stage 按 page 粒度传输
    3. PD 切换期间 pipeline bubble 为零
    4. 支持 PD + PP 2.5D 并行模式

    REQ-DIST-032 PP 与 Context Parallelism 协同

    PP 与 Ring Attention Context Parallelism 协同：PP stage 内部可进一步切分为 CP 子段处理超长序列。CP 环通信限定在同 PP stage 内，跨 stage 无额外 CP 开销。

    验收标准：
    1. PP 与 Ring Attention CP 协同
    2. PP stage 内部可切分为多个 CP 子段
    3. CP 环通信仅在同 PP stage 内发生
    4. 长序列支持 pp_size * cp_size 倍于单设备长度
    5. PP+CP 数值与单 GPU 长序列等价

    REQ-DIST-033 PP 与 SAGUARO 推测协同

    PP 与 SAGUARO 分布式推测协同：Pipeline Parallel 与 SAGUARO 分布式推测解码协同。Draft model 可在独立的 PP stage 子集上运行（利用较少层），Verify model 在完整 PP stage 上运行。

    验收标准：
    1. PP 与 SAGUARO 分布式推测协同
    2. Draft model 在独立 PP stage 或子集运行
    3. Verify model 在完整 PP stage 运行
    4. Draft/Verify 加速比 >= 1.5x
    5. PP 通信与推测通信共存正确

    REQ-DIST-034 PP 故障恢复

    Pipeline Parallel 故障恢复：Stage 级故障检测与恢复机制。Stage 心跳超时触发故障标记，故障隔离防止脏数据扩散，Stage 迁移将故障 stage 层权重迁移到存活 stage 恢复流水线。

    验收标准：
    1. PP 故障检测：stage 心跳超时 > threshold → 标记 faulted
    2. 故障隔离：faulted stage 的 KV cache 和激活被隔离
    3. Stage 迁移：将 faulted stage 层权重迁移到存活 stage
    4. 迁移后 pipeline 恢复执行，输出等价
    5. 迁移完成时间 < 30s
