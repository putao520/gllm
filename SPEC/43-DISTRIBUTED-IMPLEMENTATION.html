§0 动机与约束

§0.1 问题

gllm 当前仅支持单 GPU 推理。大模型（70B+）和 MoE 模型（A22B+）需要多 GPU 协同推理。gllm-nccl 已提供底层通信原语（AllReduce/AllGather/ReduceScatter/AllToAll/RDMA），但 gllm 侧缺少完整的分布式集成层：通信句柄生命周期、TP 权重分片、通信-计算重叠调度、PD 分离路由、KV 跨节点传输、MoE 分布式 dispatch、Context Parallelism、分布式推测解码。

§0.2 约束

  • 三仓约定: gllm → gllm-kernels → gllm-nccl 单向依赖。gllm 通过 gllm-kernels VmInstr call stub 间接使用 gllm-nccl，不直接依赖。
  • ARCH-RUST-IS-CODEGEN: 分布式通信必须 JIT 编译为 VmInstr call stub，禁止 Rust 运行时编排通信。
  • ARCH-UNIFIED-COMPILER: 分布式图构建走统一 CompilerGraph，编译器通过 GraphTopologyAnalysis 推导通信点，不前置 bool flag。
  • NO-SILENT-FALLBACK: 通信失败必须返回 Err，禁止静默降级到单 GPU。
  • ARCH-JIT-YIELDS: 通信策略顺从硬件拓扑（NVLink vs PCIe vs RDMA），不要求硬件适配代码。

    §1 Layer 0: 基础设施接线

    §1.1 REQ-DIST-001: CommHandle 生命周期管理

    REQ-DIST-001 | Layer 0 | 状态: draft

    描述: Executor 构建时根据 ParallelConfig 初始化 NCCL 通信句柄。CommHandle 封装 gllm-nccl CommHandle，提供 rank()/world_size()/is_distributed()/destroy() 方法。Drop trait 实现安全释放（先 barrier 再 destroy）。

    前置条件:

    • ParallelConfig 已配置（tp_size >= 1, rank 有效）
    • gllm-nccl feature 已启用（cfg(feature = "nccl")）
    • CUDA/RoCm runtime 已初始化

    后置条件:

    • CommHandle 持有有效的 NCCL communicator
    • rank() 返回当前进程在通信组中的 rank
    • world_size() 返回通信组总大小
    • is_distributed() 在 world_size > 1 时返回 true
    • Drop 时所有 rank 同步 barrier 后 destroy

    验收标准:

    • 单 GPU: is_distributed() == false, rank() == 0, world_size() == 1
    • 多 GPU: is_distributed() == true, rank 值域 [0, world_size)
    • Drop 无资源泄漏（ncclCommDestroy 被调用）
    • 重复 Drop 安全（幂等）

    §2 Layer 1: Tensor Parallelism

    §2.1 REQ-DIST-004: TP 权重分片

    REQ-DIST-004 | Layer 1 | 状态: draft

    描述: 按 rank 对权重张量执行分片，权重加载时执行。分片策略：

    • 列切分: QKVProj/OProj — 按 num_heads/tp_size 切分列维度，每个 rank 持有 1/tp_size 的 head
    • 行切分: GateUpProj/DownProj — GateUpProj 按行切分（每个 rank 持有 1/tp_size 行），DownProj 按列切分
    • 列切分: lm_head — 按 vocab_size/tp_size 切分列维度
    • 不分片: embedding/RMSNorm/RoPE — 所有 rank 持有完整副本

    前置条件:

    • CommHandle 已初始化（REQ-DIST-001）
    • 权重文件已加载到内存
    • ModelConfig 包含 num_heads/num_kv_heads/head_dim 信息

    后置条件:

    • 每个 rank 持有对应分片的权重子张量
    • 分片后权重 shape 正确（QKVProj: [hidden, num_heads_per_rank * head_dim]）
    • 不分片的权重在所有 rank 上完全一致

    验收标准:

    • tp_size=2 时 QKVProj 列数减半
    • tp_size=4 时 GateUpProj 行数减为 1/4
    • 所有 rank 的 embedding 权重 bit-wise 一致
    • 分片后模型 forward 数值与单 GPU 等价（在 AllReduce 聚合后）

    §2.2 REQ-DIST-005: TP AllReduce 通信

    REQ-DIST-005 | Layer 1 | 状态: draft

    描述: 每个 Transformer 层 forward 后调 comm_handle.all_reduce(Sum) 聚合部分结果。行切分层（GateUpProj→DownProj）的输出需要 AllReduce 聚合；列切分层（QKVProj→OProj）的输出在 OProj 后需要 AllReduce。通信点由 CompilerGraph 的 GraphTopologyAnalysis 自动推导，不前置 bool flag。

    前置条件:

    • TP 权重分片完成（REQ-DIST-004）

    • AllReduce(Sum) 后每个 rank 持有完整的聚合结果
    • 聚合结果与单 GPU forward 数值等价（在浮点舍入误差内）
    • 通信延迟不阻塞后续独立计算

    验收标准:

    • 2-GPU TP: AllReduce 后 logits 与单 GPU 差异 < 1e-5
    • 4-GPU TP: AllReduce 后 logits 与单 GPU 差异 < 1e-4
    • 通信点数量 = num_layers * 2（attention + FFN 各一次）

    §2.3 REQ-DIST-006: TP 列并行通信

    REQ-DIST-006 | Layer 1 | 状态: draft

    描述: ReduceScatter+AllGather 替代 AllReduce 用于列切分层，支持通信-计算重叠。列并行模式：QKVProj 列切分后各 rank 独立计算 attention → ReduceScatter 聚合 → 本地 OProj → AllGather 广播。相比 AllReduce 减少通信量 50%（每个 rank 只发送 1/tp_size 数据）。

    §3 Layer 2: 通信-计算重叠

    §3.1 REQ-DIST-007: 双缓冲重叠调度

    REQ-DIST-007 | Layer 2 | 状态: draft

    描述: ForceDoubleBuffer mk_variant 返回 DoubleBufferCluster，交替调度通信/计算 SM 分区。双缓冲策略：Layer N 的通信与 Layer N+1 的计算重叠执行。通信使用专用 SM 分区，计算使用剩余 SM 分区。两个 buffer 交替使用，消除通信等待时间。

    前置条件:

    • TP 通信已实现（REQ-DIST-005/006）
    • GPU 支持 SM 分区（ComputeCapability >= 7.0）
    • Mega-Kernel 支持多 stream 调度

    后置条件:

    • DoubleBufferCluster 包含两个交替 buffer
    • 通信和计算在时间轴上重叠
    • 每个 step 的有效通信延迟趋近于 0（完全隐藏时）

    验收标准:

    • 双缓冲模式下 step 延迟 < 串行模式延迟 * (1 + 通信/计算重叠率)
    • 重叠率 >= 70% 时有效通信延迟 < 30% 原始通信延迟
    • 数值结果与串行模式完全一致

    §3.2 REQ-DIST-008: FLUX 分解调度

    REQ-DIST-008 | Layer 2 | 状态: draft

    描述: ForceFlux mk_variant 返回 FluxDecompose，将 AllReduce 分解为 ReduceScatter + LocalCompute + AllGather 三阶段，实现更细粒度的通信-计算重叠。FLUX 策略连接 gllm-nccl flux_decompose()：ReduceScatter 完成部分数据后立即启动本地计算，不等全部数据到齐。AllGather 在本地计算完成后立即启动，不等所有 rank 完成。

    §4 Layer 3: PD 分离 + KV 分布

    §4.1 REQ-DIST-011: PD 分离角色路由

    REQ-DIST-011 | Layer 3 | 状态: draft

    描述: PdDisaggMode.Disaggregated 时，Executor 按角色路由：PrefillOnly 角色只编译 prefill 路径（无 Argmax、无采样），DecodeOnly 角色只编译 decode 路径（无长序列 prefill）。角色路由省略无关编译路径，减少 JIT 编译时间和内存占用。角色由 PdDisaggConfig.role 决定，编译时通过 GraphTopologyAnalysis 推导裁剪图。

    前置条件:

    • PdDisaggConfig 已配置（mode=Disaggregated, role=PrefillOnly/DecodeOnly）
    • CommHandle 已初始化
    • CompilerGraph 支持按角色裁剪

    后置条件:

    • PrefillOnly: 编译产物不含 Argmax/采样/stop condition VmInstr
    • DecodeOnly: 编译产物不含长序列 prefill 路径
    • 角色路由不影响数值正确性

    验收标准:

    • PrefillOnly 编译产物 VmInstr 数量 < 完整编译产物
    • DecodeOnly 编译产物 VmInstr 数量 < 完整编译产物
    • PD 分离端到端推理结果与联合模式一致
    • PdDisaggMode.Joint 时行为与单 GPU 完全一致

    §4.2 REQ-DIST-012: 跨节点 KV 传输

    REQ-DIST-012 | Layer 3 | 状态: draft

    描述: KV pages 通过 comm_handle.send()/recv() 跨节点传输。Prefill 节点完成 prefill 后将 KV pages 传输到 Decode 节点。传输支持：同步传输（阻塞等待完成）、异步传输（返回 Future，与计算重叠）、RDMA 零拷贝传输（跨节点时）。传输粒度为 KvPage，支持批量传输多个 pages。

    前置条件:

    • PD 分离角色路由已实现（REQ-DIST-011）
    • CommHandle 支持 send/recv
    • KV pages 已分配并填充数据

    后置条件:

    • 发送端 KV pages 数据完整传输到接收端
    • 接收端 KV pages 数据与发送端 bit-wise 一致
    • 异步传输完成后 Future 返回 Ok
    • RDMA 传输零拷贝（数据不经过 CPU）

    验收标准:

    • 同步传输: 接收端 KV 数据与发送端完全一致
    • 异步传输: Future 完成后数据一致
    • 批量传输: N 个 pages 一次性传输，延迟 < N * 单 page 传输延迟
    • RDMA 传输: CPU 利用率 < 5%（零拷贝验证）

    §4.3 REQ-DIST-013: KV 分布 5 模式

    REQ-DIST-013 | Layer 3 | 状态: draft

    描述: KV 分布支持 5 种模式，由 KvDistributionConfig.mode 选择：

    • Local: 所有 KV pages 存储在本地 GPU，无跨 GPU 通信（默认模式）
    • OnDemand: KV pages 按需从远端 GPU 读取，读后释放远端资源
    • Mirror: KV pages 在所有 GPU 完整镜像，内存开销最大但零通信
    • PartialHeadMirror: 按 head 维度部分镜像，每个 GPU 镜像 1/tp_size 的 head（TP 模式默认）
    • TieredCache: 三级缓存（GPU HBM → CPU DRAM → 远端 GPU），按热度自动迁移

    前置条件:

    • 分布式页路由表已构建（REQ-DIST-003）
    • CommHandle 可用
    • PagedScheduler 支持分布式页分配

    后置条件:

    • 每种模式的 KV pages 存储位置符合模式定义
    • 模式切换时路由表正确重建
    • 所有模式的 forward 数值结果一致

    验收标准:

    • Local: 零跨 GPU 通信，内存 = 单 GPU * 1
    • OnDemand: 按需通信，内存 < Mirror
    • Mirror: 零通信，内存 = 单 GPU * tp_size
    • PartialHeadMirror: 通信量 = attention 输入大小，内存 = 单 GPU * 1
    • TieredCache: 热页在 HBM，冷页在 DRAM/远端，自动迁移

    关联 Entity: ENT-DIST-KV-MODE

    关联 Dataflow: DF-DIST-004

    §5 Layer 4: MoE 分布式

    §5.1 REQ-DIST-014: 分布式 MoE

    REQ-DIST-014 | Layer 4 | 状态: draft

    描述: ExpertPlacement 定义权重分片策略，AllToAll 实现跨 GPU expert dispatch。分布式 MoE 流程：token 经 router 计算 top-k expert → AllToAll dispatch（token 发送到 expert 所在 GPU）→ expert 计算 → AllToAll merge（结果返回原 GPU）。ExpertPlacement 策略：

    • RoundRobin: expert 均匀分配到各 GPU
    • HotCold: 热 expert 镜像到多 GPU，冷 expert 独占单 GPU

    §6.1 REQ-DIST-016: Ring Attention Context Parallelism

    REQ-DIST-016 | Layer 5 | 状态: draft

    REQ-DIST-017 | Layer 5 | 状态: draft

    ID: CF-DIST-005

    • 每轮: 本地 Q × 当前 KV 块 → 累积 partial attention

    SAGUARO 三阶段:

    • Stage 1: Draft GPU 生成 N 个候选 token
    • Stage 2: send(candidates) → Verify GPU 并行验证 → send(verify_results)
    • Stage 3: Draft GPU 接收验证结果 → 更新状态 → 继续生成
    • 流水线: Stage 1(N+1) ‖ Stage 2(N) 重叠
