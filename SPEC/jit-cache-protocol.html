1. 核心铁律

> JIT 编译只允许在两个时间点发生：

> 1. 模型加载时 — 静态形状已知，auto_graph 构建 CompilerGraph，通过 MegaKernelExecutor 一次性编译为单一 Mega-Kernel 机器码。

> 2. Autotuning 窗口期 — 首次推理的前 3-5 秒，执行微基准测试搜索最优 tile 配置。

> 推理热路径中禁止出现任何编译行为。

> CPU 与 GPU 后端共享完全一致的全模型执行图。

2. 缓存架构

2.1 缓存键 (Cache Key)

缓存键 = 模型结构签名，不包含任何运行时动态维度：

```rust

ModelArchKey {

model_id: String, // "meta-llama/Llama-3-70B" 或模型路径 hash

backend: BackendKind, // Cuda / Hip / Metal / Cpu

// ... 其他全模型级静态参数

}

```

L3 磁盘缓存文件名格式:

```

{model_hash}_{backend}.bin

```

2.2 缓存粒度 (MegaKernelExecutor)

> 铁律：缓存粒度为 全模型单体 Mega-Kernel (Whole-Model Mega-Kernel)。

> gllm 彻底抛弃基于 GraphType 分离算子或分离网络层的缓存体制（摒弃诸如 FusedAttentionLayer、FusedFfnLayer 的碎块化管理）。auto_graph 从 tensor names 构建 CompilerGraph，MegaKernelExecutor 在加载期将其编译为单一 Mega-Kernel 机器码实例。

> 内存中仅存在单一共享实例。禁止任何层级、算子级的局部后备缓存。所有推理通过一次 CALL 进入预编译 Mega-Kernel 执行。

> 在线旋转矩阵归属: TurboQuant 2.0 的在线 FWHT 旋转矩阵 $R_{online}^{(l)}$ 是模型级静态常数（每层一个，不随请求变化）。它作为模型加载时的附属元数据与权重一起注入 MegaKernelExecutor，在 JIT 编译时作为立即数常量嵌入 FWHT 指令序列。不新建独立缓存。

2.3 CPU / GPU 后端统一要求

无论是 CPU 还是 GPU，模型加载时必须只生成且只缓存一条 MegaKernelExecutor 编译产物。不允许 GPU 使用一种按层编译的 HashMap，而 CPU 使用另一种独立拓扑。

所有后端通过统一的上下文环境，共同依赖底层单一 Mega-Kernel 编译产物。

4. 模型加载时编译流程

```text

Client::load_model(model_id)

→ Loader 加载权重 + config.json

→ 构造模型级的 ModelArchKey

→ 查询磁盘/内存级缓存

→ 没有命中时: auto_graph 从 tensor names 构造全网结构图

→ MegaKernelExecutor 接管进行全局统一编译（仅在加载时发生一次）

→ 将 Mega-Kernel 编译产物作为唯一缓存实体写回

→ 推理阶段通过一次 CALL 进入 Mega-Kernel 执行

```
