统一编译器架构 - 接口契约 (39A)

SPEC > 39-UNIFIED-MEGA-KERNEL > 39A-COMPILER-CONTRACT

接口契约（外部接口，读者=gllm 开发者）

> 核心定位: 本文档定义 gllm-kernels 编译器的外部接口契约，包括 compile 入口签名、输入类型（CompilerGraph）、输出类型（MegaKernelFn）、不变量和禁止规则。gllm 开发者需要遵守这些约束才能正确调用编译器。

> 与 39B 的关系: 39B-COMPILER-INTERNALS 定义编译器内部实现细节（GroupMarker、ForwardPhaseDispatch、LoweringContext、并行化策略等），读者是 gllm-kernels 开发者。

> 拆分说明: 原 SPEC 39-UNIFIED-MEGA-KERNEL 混合了外部接口与内部实现细节，本次修复按受众拆分为 39A（接口）和 39B（内部）。

§1 REQ-UMK 接口契约

§1.1 唯一编译入口 (REQ-UMK-001)

确立 InferenceCompiler::compile() 为编译器唯一公开方法，接受任意拓扑的 CompilerGraph 并产出 CompileOutput。消除所有独立编译路径（compile_graph、compile_mega_kernel_from_graph、compile_forward），1 个 Gather op 的 SG 小图、RmsNorm+Gemm 的 K 投影图、完整 35 层 decoder 图全部走同一入口。编译器零图结构假设，模型形态差异 100% 由图拓扑编码。

impl InferenceCompiler {
    pub fn compile(
        &mut self,
        graph: CompilerGraph,
        config: &CompileConfig,
    ) -> Result<CompileOutput, InferenceError>;
}
  • 一个函数。不存在 compile_graph、compile_mega_kernel_from_graph、compile_forward 等独立路径
  • 任意 CompilerGraph：1 个 Gather op 的 SG 小图、RmsNorm+Gemm 的 K 投影图、完整 35 层 decoder 图——全部走这个入口
  • CompileConfig 携带编译器需要的非图推导参数（max_seq_len、SG/hook 配置等），但不携带任何图结构假设。OutputMode 是 BUILD 阶段策略参数，不传给编译器
REQ-UMK-001 compile() 唯一编译入口 — 所有图走同一管线产出 MegaKernelFn
  1. compile() 是 InferenceCompiler 的唯一公开编译方法
  2. grep -rn "compile_graph|compile_forward|compile_mega_kernel_from_graph" src/compiler/ 无结果（除 compile() 实现内部）
  3. 任意 CompilerGraph（1-op 小图 / encoder / decoder）均通过 compile() 编译成功
  4. 无 is_simple 双路径分攧——1-op 小图和 35 层 decoder 图走完全相同的 emit 逻辑
  5. compile_simple_graph_internal 函数不存在
§2.1 唯一函数签名：MegaKernelFn (REQ-UMK-002)

所有编译产物（1-op 小图、encoder、decoder、vision、audio）使用 MegaKernelFn ABI：

pub type MegaKernelFn = unsafe extern "C" fn(
    *const u32,   // arg 0:  input_ids_ptr
    *const u8,    // arg 1:  weight_blob_ptr
    *mut u8,      // arg 2:  kv_cache_ptr
    *const u32,   // arg 3:  positions_ptr
    *const u8,    // arg 4:  aux_ptr
    usize,        // arg 5:  batch_size
    usize,        // arg 6:  prompt_len
    *mut u8,      // arg 7:  scratchpad_ptr
    *mut u32,     // arg 8:  output_tokens_ptr
    usize,        // arg 9:  temperature_u32
    usize,        // arg 10: top_k
    usize,        // arg 11: top_p_u32
    usize,        // arg 12: max_new_tokens
    usize,        // arg 13: eos_token_id
    *const u8,    // arg 14: hook_ctx_ptr
    *mut u8,      // arg 15: telemetry_ptr
    usize,        // arg 16: session_position
    *const u8,    // arg 17: fused_hidden_ptr
    usize,        // arg 18: num_mm_tokens
    *const u8,    // arg 19: callback_table_ptr
    *const u32,   // arg 20: page_table_ptr
    *const u8,    // arg 21: batch_ctx_ptr
) -> usize;
REQ-UMK-002 MegaKernelFn 统一 ABI — 所有编译产物使用 22 参数签名
  1. entry_fn() 类型统一为 MegaKernelFn
  2. 不存在 CompiledLayerFn 引用
  3. 所有编译产物使用 22-param ABI
§1.3 Phase 3 必须图结构无关 -- 详细重构规范 (REQ-UMK-003) 当前（违反设计）：compile_mega_kernel_vm() 硬编码 Phase 0-8： 
Phase 0: prologue + weight_ptr + scratchpad
Phase 1: embed (Gather op)
Phase 2-3: prefill (N 层循环)
Phase 4-5: decode (N 层循环 + 采样)
Phase 6: Argmax + StoreToken
Phase 7: CheckStopCondition + 循环回边
Phase 8: epilogue + ret

编译器假设图一定有 embed→layers→lm_head→sampling→token_store→EOS_check 结构。

encoder/SG 小图不符合此假设 → 需要第二条路径 compile_graph()。

目标（正确设计）：编译器不假设图结构，从 plan.groups 迭代发射： 
Phase 0: prologue（栈帧、weight_ptr、scratchpad_ptr、output_ptr）
Phase 0.5: ForwardPhaseDispatch（仅多步生成图存在）
for group in plan.groups:
    emit_group(group)  <-- 图有什么算子就发射什么
Phase N: epilogue + ret
REQ-UMK-003 GraphTopologyAnalysis 从图拓扑推导一切 — 替代 bool 参数和前置假设
  1. compile_mega_kernel_vm 无硬编码 Phase 编号注释——Phase 编号替换为描述性名称（如"ABI prologue"/"outer loop setup"/"采样管线"等）
  2. 无 is_encoder 参数——所有图结构属性从 GraphTopologyAnalysis 推导
  3. 无 lm_head label 搜索逻辑——grep -rn 'lm_head' src/compiler/codegen/vm/mega_kernel_emit.rs 无结果（注释除外）
  4. 无 embed→layers→lm_head 权重布局假设——权重通过 named_offsets 映射表寻址
  5. 无 vocab_size 作为 compile_mega_kernel_vm 参数——从 topology.vocab_size 获取
  6. logits 重定向通过 topology.logits_producer_op_idx 驱动，非 label 约定
§1.5 辅助小图（SG Level Keys 预计算、Vision/Audio 编码）(REQ-UMK-004)

SG 的 EmbedLookupOnlyGraph（Gather-only）、KProjOnlyGraph（RmsNorm+Gemm）等辅助图也通过同一个 compile() 入口编译，使用同一个 MegaKernelFn ABI。

调用时：

  • input_ids_ptr → 指向索引数据（u32 token IDs 或 u32 编码的索引）
  • weight_blob_ptr → 指向小图独立的权重 blob
  • kv_cache_ptr → NULL
  • max_new_tokens → 0
  • 其余采样/KV/session 参数 → NULL/0

小图编译产物与完整模型编译产物拥有相同的 ABI 签名、相同的 prologue/epilogue 结构、相同的 CompiledLayer 存储。区别仅在于图里的 ops 不同 → JIT 机器码不同。

REQ-UMK-004 SG 辅助图统一编译 — Gather/RmsNorm+Gemm 小图走同一 compile() 入口
  1. grep -rn "compile_graph|InferenceCompiler::new.*compile_graph" src/semantic_gatekeeper/ src/compat/ 无结果
  2. SG 小图与完整模型走同一 compile() 入口
  3. 小图编译产物使用 MegaKernelFn ABI
§4.1 图输出 → scratchpad output region (REQ-UMK-006)

编译器将 CompilerGraph.outputs[0] 重定向到 scratchpad + logits_scratch_offset（紧接在所有中间 tensor 和 RoPE cache 之后）。调用方从该区域读取结果。

  • Decoder：lm_head 输出 logits → scratchpad 中间区域（prefill）或直接采样
  • Encoder：MeanPool/Classify/Rerank 输出 → scratchpad 输出区域，调用方读取
  • SG 小图：Gather/Gemm 输出 → scratchpad 输出区域，调用方读取
REQ-UMK-006 Logits 重定向至 scratchpad — 调用方从编译输出区读取结果
  1. grep -rn "logits_scratch_offset|output_tensor" src/engine/mega_kernel/ 有结果
  2. encoder 调用后从 scratchpad + logits_scratch_offset 读取结果
REQ-UMK-011 CompileConfig 定义 (REQ-UMK-011)

定义 CompileConfig 结构体，仅携带编译器所需的非图推导参数。BusinessConfig 已彻底分离——编译器不持有 BusinessConfig 引用，结构上保证编译器无法访问 output_modes 等业务字段。debug_jit 是编译器唯一直接读取的非图推导参数，因为它影响全局 codegen 行为（INT3 插入、source map 生成），无法通过 OpKind 表达。

定义 CompileConfig 结构体，仅携带编译器所需的非图推导参数（内存上界、权重偏移映射、采样配置、设备能力等）。禁止包含任何图结构假设字段（is_encoder/inference_mode/vocab_size/num_layers/hidden），这些信息全部由 CompilerGraph 拓扑自动推导。字段来源严格遵循 ARCH-DATA-FLOW-CONTRACT，每个字段都有唯一合法来源。

/// 编译配置 -- 携带编译器需要的非图推导参数。
///
/// 设计原则：CompileConfig 不携带任何图结构假设。
/// 编译器从 CompilerGraph 推导一切结构信息。
/// CompileConfig 只提供"图本身无法推导的外部输入"。
///
/// BusinessConfig 已彻底分离：编译器不持有 BusinessConfig 引用，
/// 结构上保证编译器无法访问 output_modes 等业务字段。
/// BusinessConfig 仅在 BUILD 阶段（图构建）使用，不传入编译器。
pub struct CompileConfig {
    // -- 内存上界 --
    /// Buffer 分配上界（替代 SYMDIM_MAX_SEQ_LEN 硬编码）。
    /// 来自模型配置 max_position_embeddings，不是编译器硬编码。
    /// 仅用于 scratchpad/KV buffer 分配大小，不影响循环 bound。
    pub max_seq_len: usize,
    // -- 调试 --
    /// 启用 JIT 调试输出（INT3 breakpoints + source map）。
    /// 这是编译器唯一直接读取的非图推导参数——debug_jit 影响全局
    /// codegen 行为（INT3 插入、source map 生成），无法通过 OpKind 表达。
    pub debug_jit: bool,
    // -- 异构层配置 --
    /// 异构层布局（Gemma 4 E2B 等）。None = 均匀层。
    /// 编译器通过 GprCondAction 在 JIT 内条件分支，不由 CompileConfig 驱动。
    pub hetero: Option<HeteroLayerConfig>,
}

REQ-LOWER-TOTALITY-001 Lowering Totality 编译时门控

Lowering Totality 编译时门控标准： 1. 构建集 ⊆ 可 lower 集：CompilerGraph 中每个 OpKind 必须在 ScalarOpRegistry 有注册 + auto_select 有 match arm 2. OpSemantics 拆分：MetadataNop（纯元数据 op，允许 NOP：仅 Reshape/Transpose）vs Unimplemented（编译时 Err）vs Implemented（有完整 lowering） 3. 缺少注册或 lowering = 编译失败（不是运行时 panic） 4. 实施分两阶段：Phase 1 拆分 Opaque 为 MetadataNop vs Unimplemented；Phase 2 逐个补全 Unimplemented 的 lowering 验证方法：构建时 audit_opkind_registry() + audit_auto_select_coverage() 交叉引用：REQ-OPKIND-REG-001 / REQ-OPKIND-REG-002 / REQ-BC-GRAPH-002

验收标准：
  1. 构建集 ⊆ 可 lower 集：CompilerGraph 中每个 OpKind 必须在 ScalarOpRegistry 有注册 + auto_select 有 match arm
  2. OpSemantics 拆分：MetadataNop（纯元数据 op，允许 NOP：仅 Reshape/Transpose）vs Unimplemented（编译时 Err）vs Implemented（有完整 lowering）
  3. 缺少注册或 lowering = 编译失败（不是运行时 panic）
  4. 实施分两阶段：Phase 1 拆分 Opaque 为 MetadataNop vs Unimplemented；Phase 2 逐个补全 Unimplemented 的 lowering 验证方法：构建时 audit_opkind_registry() + audit_auto_select_coverage() 交叉引用：REQ-OPKIND-REG-001 / REQ-OPKIND-REG-002 / REQ-BC-GRAPH-002

§X CompileConfig 输入集详细描述（REQ-UMK-001 扩展）

外部接口（gllm 调用方使用）

InferenceCompiler::compile(graph: &CompilerGraph, config: &CompileConfig) -> Result

外部接口仅 2 个参数：graph（要编译的算子图）+ config（编译配置）。其余 13 个依赖由 compile 内部自动构造。

内部 lowering 入口（gllm-kernels 内部使用）

compile_mega_kernel_vm(plan, graph, alloc, registry, profile, hook, buffer_layout, bottleneck_map, virtual_activation, virtual_tensor_map, layout, debug_jit, _mtp_config_removed, resource_plan, topology) -> Result

16 参数（含 plan 输出参数）。由 compile 外部接口自动构造传入，gllm 调用方不直接使用。

13 个额外输入来源

参数来源构造时机
plancompile 内部 newcompile 入口
alloccompile 内部 new（VReg allocator）compile 入口
registry: &ScalarOpRegistrygllm-kernels static global静态构造
profile: &IsaProfileDeviceProfile::detect() 推导compile 入口
hook: Option<&dyn IsaHook>config.hook 或 Noneconfig 字段
buffer_layoutBufferLayout::plan(graph) 推导compile 内部 pass
bottleneck_mapanalyze_bottlenecks(graph) 推导compile 内部 pass
virtual_activationVirtualActivationMap::build(graph) 推导compile 内部 pass
virtual_tensor_mapVirtualTensorMap::build(graph) 推导compile 内部 pass
layoutLayoutAssignment::build(graph) 推导compile 内部 pass
debug_jitconfig.debug_jitconfig 字段
resource_planResourcePlanner::plan(graph) 推导compile 内部 pass
topologyGraphTopologyAnalysis::analyze(graph) 推导compile 内部 pass

设计原则：外部接口保持简单（2 参数），内部 lowering 接受完整状态（16 参数）。两层正交，gllm 调用方无需关心 13 个内部依赖。
