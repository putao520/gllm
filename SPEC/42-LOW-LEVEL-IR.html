Low-Level IR (LIR) — TraceOp 与 VmInstr 之间的计算语义桥

SPEC > 42-LOW-LEVEL-IR

概述

> SSOT: 本文档定义 gllm-kernels JIT 编译器的 LIR 层架构。在 TraceOp(源级 SSA) 与 VmInstr(状态追踪) 之间插入结构化计算描述层。

> 交叉引用: SPEC/01-JIT-PIPELINE.html 定义四阶段管线; SPEC/03-GRAPH-IR.html 定义 CompilerGraph IR; SPEC/26-VMINSTR-RATIONALIZATION.html 定义 VmInstr 去类型化合并; SPEC/GRAPH-SHAPE-DRIVEN-MEGA-KERNEL.html 定义统一编译器架构。

1. 动机

当前 TraceOp-VmInstr 语义鸿沟确凿存在。80+ TraceOp 变体 vs 131+ VmInstr 变体，VmInstr 混杂计算表达与状态追踪。ISA 优化 pass 在 VmInstr 层执行模式匹配时缺乏计算语义上下文，导致手写 emit 函数和 ad-hoc lowering。

LIR 填补语义鸿沟，使计算语义显式化，支持结构化 pass pipeline，收窄 VmInstr 职责为纯状态追踪。

2. 需求定义

REQ-LIR-001: LIR 定位为 Compute Pattern Expression Layer

LIR 是 TraceOp(源级SSA) 与 VmInstr(状态追踪) 之间的计算语义桥。LIR 不替代 TraceOp 或 VmInstr，而是在两者之间插入结构化计算描述层。

验收标准：
  1. LIR 类型定义在 compiler/llir/ 模块
  2. LIR 位于 TraceOp lowering 和 VmInstr 构造之间
  3. 所有通过 LIR 的计算路径保留完整语义信息
  4. 现有 TraceOp 和 VmInstr 接口不变

REQ-LIR-002: LIR 采用 Typed SSA + Region + Tile 抽象

LIR 不是枚举式胖操作码，而是结构化计算描述。LirOp 携带完整类型信息，LirRegion 支持嵌套结构，LirTile 携带分块维度和步长信息。

验收标准：
  1. LirOp 携带完整类型信息
  2. LirRegion 支持嵌套结构
  3. LirTile 携带分块维度和步长信息
  4. 三个示范算子(RMS Norm/GEMM/Attention)通过 LIR 完整表达

REQ-LIR-003: AlgorithmProfile 闭源库算法模式声明式表达

闭源库算法模式不是手写 emit 函数，而是声明式参数化描述。AlgorithmProfile 类型定义完整，至少覆盖 GEMM 和 Attention 两个闭源库模式，声明式描述可被 pass pipeline 自动匹配。

验收标准：
  1. AlgorithmProfile 类型定义完整
  2. 至少覆盖 GEMM 和 Attention 两个闭源库模式
  3. 声明式描述可被 pass pipeline 自动匹配
  4. 无手写 emit 函数依赖

REQ-LIR-004: LIR 四域划分 Compute/Memory/Layout/Communication

LIR 操作按计算语义划分为四个域：Compute(计算)、Memory(内存)、Layout(布局)、Communication(通信)。各域独立 pass pipeline，域间依赖通过显式接口表达。Communication 域为 nccl feature-gated，非分布式场景下不激活。

验收标准：
  1. 四个域类型定义完整（Compute/Memory/Layout/Communication）
  2. 每个 LirOp 明确属于唯一域
  3. 各域有独立 pass pipeline
  4. 域间依赖通过显式接口表达
  5. 三个示范算子正确归域
  6. Communication 域在 nccl feature 启用时激活，非分布式场景下零开销

REQ-LIR-005: VmInstr 职责收窄为 State Tracking Only

计算语义上移到 LIR，VmInstr 只记录寄存器/内存/控制流状态。迁移后 VmInstr 变体不再承载计算语义，计算语义通过 LIR→VmInstr 映射保持完整。

验收标准：
  1. VmInstr 变体不再承载计算语义
  2. 计算语义通过 LIR→VmInstr 映射保持完整
  3. 现有 VmInstr 接口向后兼容
  4. 迁移后 VmInstr 变体数量减少

REQ-LIR-006: ISA 优化 pass 在 LIR 层执行

模式匹配和重写发生在有计算语义的层。融合 pass 在 LIR Compute 域执行，常量折叠和死代码消除在 LIR 层执行，VmInstr 层不再执行任何计算语义优化。

验收标准：
  1. 融合 pass 在 LIR Compute 域执行
  2. 常量折叠在 LIR 层执行
  3. 死代码消除在 LIR 层执行
  4. VmInstr 层不再执行任何计算语义优化
  5. 优化结果通过 LIR→VmInstr 映射保持正确

REQ-LIR-007: Phase 1 — 定义 LIR 类型 + 三个示范算子

Phase 1 创建 compiler/llir/ 模块，定义核心类型(LirOp/LirRegion/LirTile/LirDomain)，三个示范算子(RMS Norm/GEMM/Attention)通过 LIR 完整表达，LIR 可 lowering 到 VmInstr。

验收标准：
  1. compiler/llir/ 模块创建
  2. 核心类型定义完整
  3. 三个示范算子通过 LIR 完整表达
  4. LIR 可 lowering 到 VmInstr
  5. cargo check 通过

REQ-LIR-008: Phase 2 — VmInstr 职责收窄

Phase 2 识别并列出承载计算语义的 VmInstr 变体，每个变体迁移后 VmInstr 只保留状态字段，所有迁移后的路径通过数值对齐测试。

验收标准：
  1. 识别并列出承载计算语义的 VmInstr 变体
  2. 每个变体迁移后 VmInstr 只保留状态字段
  3. 所有迁移后的路径通过数值对齐测试
  4. VmInstr 变体数量净减少

REQ-LIR-009: Phase 3 — pass pipeline 迁移

Phase 3 将融合 pass 迁移到 LIR Compute 域，内存布局优化 pass 迁移到 LIR Memory 域，常量折叠和死代码消除迁移到 LIR 层，四域 pass pipeline 独立调度。Communication 域 pass（通信融合、chunk 级重叠调度、量化压缩）在 nccl feature 启用时参与 pipeline。

验收标准：
  1. 融合 pass 迁移到 LIR Compute 域
  2. 内存布局优化 pass 迁移到 LIR Memory 域
  3. 常量折叠和死代码消除迁移到 LIR 层
  4. 四域 pass pipeline 独立调度
  5. Communication 域 pass 在 nccl feature 启用时参与 pipeline
  6. 优化结果与迁移前数值一致

REQ-LIR-010: Phase 4 — 闭源库对齐

Phase 4 用 AlgorithmProfile 描述闭源库 GEMM 和 Attention 算法模式，AlgorithmProfile 可被 pass pipeline 自动匹配，所有手写 emit_xxx 函数替换为声明式描述。

验收标准：
  1. 闭源库 GEMM 用 AlgorithmProfile 描述
  2. 闭源库 Attention 用 AlgorithmProfile 描述
  3. AlgorithmProfile 可被 pass pipeline 自动匹配
  4. 所有手写 emit_xxx 函数替换为声明式描述
  5. 闭源库路径与非闭源库路径数值一致

REQ-LIR-011: LIR 模块位置 compiler/llir/

LIR 在编译管线中的位置：TraceOp → (auto_lower_trace) → LIR → (lir_to_vminstr) → VmInstr → ISA Lowering。LIR 不修改现有 TraceOp 定义和 VmInstr 定义，通过新增 lowering pass 接入管线。

验收标准：
  1. compiler/llir/ 模块创建
  2. LIR 不修改现有 TraceOp 定义
  3. LIR 不修改现有 VmInstr 定义
  4. LIR 通过新增 lowering pass 接入管线
  5. 现有编译路径可通过 feature flag 切换

5. Interface 定义

IF-LIR-001: LirPassPipeline 契约

四域独立 pass pipeline 接口。域间通过显式依赖排序：Layout→Compute→Memory→Communication。每个域有独立的 pass 注册和执行调度。Communication 域 nccl feature-gated。

输入

  • LirProgram — LIR 程序（包含四域 LirOp 集合）
  • DeviceProfile — 硬件能力约束
  • pass_config: PassConfig — pass 配置参数

输出

  • LirProgram — 优化后的 LIR 程序
  • pass_log: Vec<PassLogEntry> — pass 执行日志

错误类型

  • LirError::DomainDependencyCycle — 域间依赖循环
  • LirError::PassExecutionFailed — pass 执行失败
  • LirError::InvalidDomainAssignment — 操作域归属无效

7. Decision 定义

DEC-LIR-001: LIR 定位为 Compute Pattern Expression Layer（新增层而非替换）

LIR 作为新增层插入 TraceOp 与 VmInstr 之间，不替代现有 TraceOp 或 VmInstr。现有接口不变，LIR 通过新增 lowering pass 接入管线。

方案选项

  • Option A: 扩展 TraceOp 语义（破坏现有 SSA 纯净性）
  • Option B: 扩展 VmInstr 语义（加剧计算/状态混杂）
  • Option C: 新增 LIR 层（chosen — 保持现有层不变，填补语义鸿沟）

选定方案

Option C：新增 LIR 层，保持 TraceOp 和 VmInstr 接口不变

理由

新增层最小化迁移风险，保持现有编译路径兼容。LIR 填补语义鸿沟而不破坏已有架构。符合 OCP（开放封闭原则）。

DEC-LIR-002: LIR 采用 Typed SSA + Region + Tile 抽象（非枚举式胖操作码）

LIR 操作不是枚举式胖操作码（类似 VmInstr 的 131+ 变体），而是结构化计算描述。LirOp 携带类型信息，LirRegion 表达嵌套，LirTile 表达分块。

方案选项

  • Option A: 枚举式胖操作码（类似 VmInstr 扩展，变体爆炸）
  • Option B: Typed SSA + Region + Tile 结构化抽象（chosen — 可组合、可扩展）

选定方案

Option B：Typed SSA + Region + Tile 结构化抽象

理由

结构化抽象避免变体爆炸，支持组合式扩展。Region 表达循环/条件嵌套，Tile 表达分块语义，两者正交组合覆盖所有计算模式。与 MLIR Region/Tile 设计理念一致。

DEC-LIR-003: AlgorithmProfile 声明式表达闭源库算法模式（替代手写 emit）

闭源库算法模式用 AlgorithmProfile 声明式描述，替代手写 emit_xxx 函数。声明式描述可被 pass pipeline 自动匹配和调度。

方案选项

  • Option A: 保留手写 emit 函数（维护成本高，不可组合）
  • Option B: AlgorithmProfile 声明式表达（chosen — 可匹配、可组合、可测试）

选定方案

Option B：AlgorithmProfile 声明式表达

理由

声明式表达将闭源库调用从命令式代码提升为数据，pass pipeline 可自动匹配和调度。消除手写 emit 函数的维护负担，支持多闭源库统一表达。

DEC-LIR-004: LIR 四域划分 Compute/Memory/Layout/Communication（各域独立 pass pipeline）

LIR 操作按计算语义划分为 Compute(计算)、Memory(内存)、Layout(布局)、Communication(通信) 四个域。各域有独立 pass pipeline，域间通过显式依赖排序。Communication 域为 nccl feature-gated。

方案选项

  • Option A: 单一 pass pipeline（所有优化混在一起，依赖隐式）
  • Option B: 两域划分 Compute/Memory（Layout 归入 Memory，语义不清晰）
  • Option C: 三域划分 Compute/Memory/Layout（语义清晰，但通信语义无归属）
  • Option D: 四域划分 Compute/Memory/Layout/Communication（chosen — 通信语义独立，nccl feature-gated 零开销）

选定方案

Option D：四域划分 Compute/Memory/Layout/Communication

理由

Layout 变换（转置/重排）与 Memory 操作（加载/存储）语义不同，独立域使 pass 更聚焦。Communication（AllReduce/AllToAll/P2P）与 Compute/Memory 语义正交，独立域使通信优化 pass（融合/重叠/压缩）更聚焦。域间依赖显式化：Layout→Compute→Memory→Communication，避免隐式依赖导致的 pass 执行顺序 bug。Communication 域 nccl feature-gated，非分布式场景下不激活、零开销。

DEC-LIR-005: VmInstr 职责收窄为 State Tracking Only

计算语义上移到 LIR，VmInstr 只记录寄存器/内存/控制流状态。迁移后 VmInstr 变体数量减少，职责单一化。

方案选项

  • Option A: 保持 VmInstr 现状（计算/状态混杂持续）
  • Option B: VmInstr 职责收窄为 State Tracking Only（chosen — 职责单一化）

选定方案

Option B：VmInstr 职责收窄为 State Tracking Only

理由

VmInstr 混杂计算表达与状态追踪导致 ISA 优化 pass 缺乏语义上下文。职责收窄后 VmInstr 变体减少，ISA lowering 更简单，优化 pass 在 LIR 层有完整语义信息。

DEC-LIR-006: ISA 优化 pass 在 LIR 层执行

模式匹配和重写发生在有计算语义的 LIR 层，而非缺乏语义的 VmInstr 层。VmInstr 层只做 ISA 指令选择，不做计算语义优化。

方案选项

  • Option A: 优化 pass 继续在 VmInstr 层执行（缺乏语义上下文）
  • Option B: 优化 pass 迁移到 LIR 层执行（chosen — 有完整语义上下文）

选定方案

Option B：优化 pass 迁移到 LIR 层执行

理由

LIR 层有完整计算语义，模式匹配更精确。VmInstr 层缺乏语义上下文，优化 pass 只能做局部模式匹配，容易遗漏跨指令优化机会。

DEC-LIR-007: 四阶段渐进迁移策略

LIR 采用四阶段渐进迁移：Phase 1 定义类型+示范算子 → Phase 2 VmInstr 职责收窄 → Phase 3 pass pipeline 迁移 → Phase 4 闭源库对齐。每阶段独立可验证。

方案选项

  • Option A: 一次性全量迁移（风险高，回退困难）
  • Option B: 四阶段渐进迁移（chosen — 每阶段独立可验证，风险可控）

选定方案

Option B：四阶段渐进迁移

理由

渐进迁移每阶段有明确验收标准，失败可回退到上一阶段。Phase 1 建立类型基础，Phase 2 收窄 VmInstr，Phase 3 迁移优化，Phase 4 对齐闭源库。依赖关系清晰，并行度可控。

DEC-LIR-008: LIR 新增层而非替换现有层

LIR 作为新增层插入编译管线，不替换 TraceOp 或 VmInstr。现有编译路径可通过 feature flag 切换，渐进迁移。

方案选项

  • Option A: 替换 TraceOp 或 VmInstr（破坏现有接口，迁移风险高）
  • Option B: 新增层 + feature flag 切换（chosen — 兼容现有路径，渐进迁移）

选定方案

Option B：新增层 + feature flag 切换

理由

新增层保持现有编译路径完全兼容，feature flag 允许渐进启用 LIR。迁移过程中两条路径并存，数值对齐测试可验证 LIR 路径正确性。完全迁移后 feature flag 可移除。

8. Dataflow 定义

DF-LIR-001: TraceOp → LIR lowering

TraceOp SSA 序列通过 lir_lower 变换为 LirOp 序列。ComputePattern 自动映射到 LirOp.op_kind 结构化变体。

契约

  • from: TraceOp SSA 序列 (auto_lower_trace 输出)
  • to: LirProgram (LirOp 集合 + LirRegion + LirTile)
  • transform: lir_lower(TraceOp) → LirOp
  • direction: sync（编译时一次性）

DF-LIR-002: LIR 四域 pass pipeline

LIR 四域 pass pipeline 按依赖顺序执行：Layout→Compute→Memory→Communication。各域独立 pass pipeline，域间通过显式依赖排序。Communication 域在 nccl feature 启用时参与 pipeline。

契约

  • from: LirProgram (原始)
  • to: LirProgram (优化后)
  • transform: layout_passes → compute_passes → memory_passes → comm_passes (nccl)
  • direction: sync（编译时一次性）

DF-LIR-003: LIR → VmInstr 映射

优化后的 LirProgram 通过 lir_to_vminstr 映射为 VmInstr 序列。VmInstr 只记录寄存器/内存/控制流状态，计算语义已在 LIR 层处理。

契约

  • from: LirProgram (优化后)
  • to: Vec<VmInstr>
  • transform: lir_to_vminstr(LirProgram) → Vec<VmInstr>
  • direction: sync（编译时一次性）

DF-LIR-004: AlgorithmProfile 匹配数据流

LirOp 与 AlgorithmProfile 匹配，选择闭源库算法模式。匹配成功则使用闭源库路径，匹配失败则使用默认 JIT 路径。

契约

  • from: LirOp + AlgorithmProfile 库
  • to: AlgorithmProfile | Default
  • transform: match_profile(LirOp, profiles) → AlgorithmProfile | Default
  • direction: sync（编译时一次性）

9. Controlflow 定义

CF-LIR-001: LIR pass pipeline 域间执行顺序

四域 pass pipeline 按固定顺序执行：Layout→Compute→Memory→Communication。Layout 变换先执行（影响后续计算和内存布局），Compute 优化次之（依赖 Layout 结果），Memory 优化再次（依赖 Compute 结果确定访问模式），Communication 优化最后（依赖 Compute/Memory 结果确定通信模式和缓冲区布局）。Communication 域 nccl feature-gated，非分布式场景下跳过。

CF-LIR-002: 四阶段渐进迁移控制流

四阶段按序执行：Phase 1(类型定义+示范算子) → Phase 2(VmInstr 职责收窄) → Phase 3(pass pipeline 迁移) → Phase 4(闭源库对齐)。每阶段有独立验收标准，前阶段通过后才进入下一阶段。

10. 编译管线位置

LIR 在编译管线中的位置：

TraceOp (源级 SSA)
    │
    ▼ auto_lower_trace
LirOp (计算语义桥)
    │
    ├─ Layout Domain passes
    ├─ Compute Domain passes
    ├─ Memory Domain passes
    └─ Communication Domain passes (nccl)
    │
    ▼ lir_to_vminstr
VmInstr (状态追踪)
    │
    ▼ ISA Lowering
Machine Code

关键约束：

  • LIR 不修改现有 TraceOp 定义
  • LIR 不修改现有 VmInstr 定义
  • LIR 通过新增 lowering pass 接入管线
  • 现有编译路径可通过 feature flag 切换

12. Communication Domain — 分布式通信 IR 表达

LIR Communication Domain 表达分布式通信语义，与现有 Compute/Memory/Layout 三域并列为第四域。Communication 域为 nccl feature-gated：非分布式场景下不激活，零开销。

REQ-LIR-012: LIR Communication Domain

Communication Domain 作为 LIR 第四域，处理分布式通信语义（AllReduce/AllToAll/P2P 通信、通信-计算重叠调度）。Communication 域有独立 pass pipeline，在 Layout→Compute→Memory 之后执行。域内 pass 三类：通信融合（相邻 AllReduce 合并）、通信调度（chunk 级重叠）、通信优化（量化压缩）。

验收标准：
  1. Communication 域类型定义完整，属于 LirDomain::Communication
  2. Communication 域有独立 pass pipeline，包含三类 pass
  3. Communication 域在 nccl feature 启用时激活，非分布式场景下不参与 pipeline
  4. Communication 域 pass pipeline 在 Memory 域之后执行
  5. Communication 域 LirOp 可 lowering 到 AllReduceChunk/P2pPageFetch/RdmaPageFetch 等 VmInstr
  6. Communication 域与 Compute/Memory 域的交互通过显式依赖接口表达

REQ-LIR-013: Pipeline Parallel (PP2) IR 表达

LIR 表达 2-stage 流水线并行（PP2）的通信语义。PP2 将模型按层切分为两个 stage，1F1B 调度实现通信-计算重叠：stage 0 计算 micro-batch N+1 时，stage 1 通信 micro-batch N 的激活。

核心 IR 元素：

  • PipelineStage{stage_id, micro_batch_count, forward_ops, backward_ops} — 流水线阶段定义
  • StageSend{dst_stage, buffer} — 阶段间激活发送（对应 P2pPageFetch send 端）
  • StageRecv{src_stage, buffer} — 阶段间激活接收（对应 P2pPageFetch recv 端）
  • StageBoundary{stage_id, sync_kind} — 阶段边界同步点（Barrier/Signal）
  • ActivationBuffer{stage, micro_batch, dtype, shape} — 阶段间激活缓冲区
验收标准：
  1. PipelineStage/StageSend/StageRecv/StageBoundary/ActivationBuffer 定义为 Communication 域 LirOp 变体
  2. PP2 1F1B 调度控制流可通过 StageBoundary 序列表达
  3. StageSend/StageRecv lowering 到 P2pPageFetch/RdmaPageFetch VmInstr
  4. ActivationBuffer 缓冲区布局由 Memory 域 pass 决定，Communication 域只引用
  5. 单机场景（无分布式配置）下 PP2 IR 不生成，零开销

REQ-LIR-014: Communication-Compute Overlap (MC2) IR 表达

LIR 表达通信-计算双流并发（MC2）的 IR 语义。MC2 核心是 AllReduceChunk 的 chunk 级重叠：通信流处理 chunk N 的 AllReduce 时，计算流处理 chunk N-1 的后续计算。

对应 gllm 的 OverlapHint 五变体：Auto/PreferOverlap/PreferIsolated/ForceDoubleBuffer/ForceFlux。

核心 IR 元素：

  • CommComputeOverlap{comm_ops, compute_ops, chunk_partition, sync_point} — 通信-计算重叠区域
  • ChunkPartition{total_chunks, chunk_id, chunk_size} — chunk 分割描述
  • DualStreamBarrier{kind} — 双流同步屏障（CommDone/ComputeDone/BothDone）
  • OverlapRegion{comm_instrs, compute_instrs, overlap_hint} — 重叠调度单元
验收标准：
  1. CommComputeOverlap/ChunkPartition/DualStreamBarrier/OverlapRegion 定义为 Communication 域 LirOp 变体
  2. OverlapHint 五变体映射到 OverlapRegion 重叠策略
  3. CommComputeOverlap lowering 到 AllReduceChunk + DualStreamBarrier VmInstr 组合
  4. ChunkPartition 参数从 DistributedConfig.CommConfig 推导
  5. PreferIsolated 模式下通信与计算不重叠，生成顺序执行的 VmInstr 序列

REQ-LIR-015: Unified Expert Parallelism (UNIEP) IR 表达

LIR 表达统一专家并行（UNIEP）的通信语义。UNIEP 核心是 AllToAll dispatch/combine 融合：dispatch 阶段按 expert_map 将 token 路由到目标 rank，combine 阶段按 token_map 收集专家输出。

对应 gllm 的 AllToAllStrategy（NvlinkAllToAll/RdmaAllToAll）+ ExpertPlacement。

核心 IR 元素：

  • ExpertShard{expert_id, ep_rank, placement, weight_layout} — 专家分片描述
  • AllToAllDispatch{tokens, expert_map, strategy} — 专家调度 dispatch
  • AllToAllCombine{expert_outputs, token_map, strategy} — 专家输出 combine
  • ExpertPlacement{expert_id, rank, device} — 专家放置策略
验收标准：
  1. ExpertShard/AllToAllDispatch/AllToAllCombine/ExpertPlacement 定义为 Communication 域 LirOp 变体
  2. AllToAllStrategy 映射到 AllToAllDispatch/AllToAllCombine 的 strategy 参数
  3. AllToAllDispatch/AllToAllCombine lowering 到 VmInstr AllReduceChunk（AllToAll 模式）组合
  4. ExpertShard 放置策略从 MoeDistributedConfig 推导
  5. 非 MoE 模型不生成 UNIEP IR，零开销

REQ-LIR-016: Communication Domain pass pipeline

Communication 域独立 pass pipeline 包含三类 pass，按序执行：

  1. 通信融合 pass（CommFusionPass）：相邻 AllReduce 合并、冗余通信消除、通信-通信融合
  2. 通信调度 pass（CommSchedulePass）：chunk 级重叠调度（MC2）、OverlapHint 策略映射、双流屏障插入
  3. 通信优化 pass（CommOptimizePass）：量化压缩（FP8/INT8 AllReduce）、拓扑感知路由（NVLink 优先于 RDMA）、通信缓冲区复用
验收标准：
  1. 三类 pass 定义完整，按序执行：CommFusionPass → CommSchedulePass → CommOptimizePass
  2. CommFusionPass 可合并相邻同 dtype 同 op 的 AllReduce
  3. CommSchedulePass 根据 OverlapHint 生成重叠/非重叠 VmInstr 序列
  4. CommOptimizePass 可将 AllReduce 压缩为低精度通信
  5. 三类 pass 均为 nccl feature-gated，非分布式场景下不注册

REQ-LIR-017: Tensor Parallelism (TP) IR 表达

LIR 表达张量并行（TP）的通信语义。TP 将权重按列/行切分到多 rank，每个 rank 独立计算部分结果，AllReduce/ReduceScatter+AllGather 聚合。LIR 表达切分策略与同步聚合点。

来源：SGLang ✅ / vLLM ✅ / gllm ✅（CompilerGraph + DeviceProfile 驱动 TP 切分）

核心 IR 元素：

  • WeightShard{rank, world_size, axis} — 权重切分描述（axis: Row/Col）
  • AllReduceSync{reduce_op, dtype} — AllReduce 同步聚合（Sum/Max/Min）
  • ReduceScatterSlice{chunk_size, dtype} — ReduceScatter 切片聚合
验收标准：
  1. WeightShard/AllReduceSync/ReduceScatterSlice 定义为 Communication 域 LirOp 变体
  2. AllReduceSync lowering 到 AllReduceChunk VmInstr
  3. WeightShard → 权重加载时按 rank 偏移，编译时 bake 进地址计算
  4. TP 粒度与 CompilerGraph 的 Op 切分一致
  5. 单 rank 场景（world_size=1）TP IR 不生成，零开销

REQ-LIR-018: Context Parallelism (CP) IR 表达

LIR 表达上下文并行（CP）的通信语义。CP 将长序列按 chunk 分配到多 rank，Ring Attention 每步传递 KV chunk，实现超长上下文推理。通信模式为环形 Send/Recv。

来源：vLLM ✅（Ring Attention）/ SGLang ❌ / gllm ❌（预留）

核心 IR 元素：

  • RingAttention{chunk_idx, world_size, kv_send_rank, kv_recv_rank} — Ring Attention 通信步骤
  • ContextSync{sync_point, barrier_id} — 上下文同步屏障
验收标准：
  1. RingAttention/ContextSync 定义为 Communication 域 LirOp 变体
  2. RingAttention lowering 到 P2pCopy + CommBarrier VmInstr 组合
  3. ContextSync lowering 到 CommBarrier VmInstr
  4. CP 仅在 seq_len > max_seq_per_rank 时激活
  5. 非分布式场景零开销（CP IR 不生成）

REQ-LIR-019: PD Disaggregation IR 表达

LIR 表达 Prefill-Decode 分离架构的 KV 迁移语义。PD 分离架构中，Prefill 节点计算 KV cache 后通过 RDMA/NVLink 传输到 Decode 节点。LIR 表达 KV 迁移的语义（不表达传输协议细节，那是 CommInstr 层的职责）。

来源：SGLang ✅ / vLLM ✅ / gllm ✅（PdDisaggConfig）

核心 IR 元素：

  • KvTransferOut{session_id, layer_range, page_indices} — KV cache 输出迁移
  • KvTransferIn{session_id, layer_range, page_indices} — KV cache 输入接收
  • PdHandshake{role, session_id} — PD 节点握手同步（role: Prefill/Decode）
验收标准：
  1. KvTransferOut/KvTransferIn/PdHandshake 定义为 Communication 域 LirOp 变体
  2. KvTransferOut lowering 到 AsyncCopy + AsyncCopyCommit CommInstr
  3. KvTransferIn lowering 到 AsyncWaitGroup + 数据拷贝 CommInstr
  4. PdHandshake lowering 到 AllReduce 状态同步 CommInstr
  5. PdDisaggMode::Collocated 时不生成 PD IR
  6. layer_range 支持 PP 场景下部分层迁移

REQ-LIR-020: Quantized Communication IR 表达

LIR 表达通信前量化压缩与通信后反量化恢复的语义。通信量化（FP32→INT8/FP8，4x 带宽节省）+ Delta Encoding（差分编码+位打包，进一步压缩）。LIR 表达压缩策略，CommInstr 层执行实际量化操作。

来源：gllm ✅（QuantizedComm in gllm-nccl）/ SGLang ❌ / vLLM ❌

核心 IR 元素：

  • CommQuantize{src_dtype, dst_dtype, scale, count} — 通信前量化压缩
  • CommDequantize{src_dtype, dst_dtype, scale, count} — 通信后反量化恢复
  • CommDeltaEncode{bits_per_elem, prev, count} — Delta 差分编码
  • CommDeltaDecode{bits_per_elem, prev, count} — Delta 差分解码
  • CommBitPack{bits_per_elem, count} — 位打包压缩
  • CommBitUnpack{bits_per_elem, count} — 位解包恢复
验收标准：
  1. CommQuantize/CommDequantize/CommDeltaEncode/CommDeltaDecode/CommBitPack/CommBitUnpack 定义为 Communication 域 LirOp 变体
  2. CommQuantize lowering 到 CommInstr::Quantize
  3. CommDequantize lowering 到 CommInstr::Dequantize
  4. CommDeltaEncode lowering 到 CommInstr::DeltaEncode
  5. CommBitPack lowering 到 CommInstr::BitPack
  6. CommCompressHint 驱动压缩策略选择
  7. 量化精度损失由 SPEC 10-QUALITY 约束

REQ-LIR-021: FLUX Decomposition IR 表达

LIR 表达 FLUX 分解的通信语义。FLUX 将标准 AllReduce 分解为 ReduceScatter → LocalCompute → AllGather 三阶段。ReduceScatter 完成后每个 rank 持有完全归约的 home chunk，立即开始本地计算（与 AllGather 并行），实现 ≥90% 通信-计算重叠。

来源：gllm ✅（flux_lower.rs in gllm-nccl）/ SGLang ❌ / vLLM ❌

核心 IR 元素：

  • FluxReduceScatter{chunk_count, dtype, reduce_op} — FLUX ReduceScatter 阶段
  • FluxLocalCompute{home_chunk_idx} — FLUX 本地计算阶段（绑定 Compute 域 LirOp）
  • FluxAllGather{chunk_count, dtype} — FLUX AllGather 阶段
验收标准：
  1. FluxReduceScatter/FluxLocalCompute/FluxAllGather 定义为 Communication 域 LirOp 变体
  2. FluxReduceScatter lowering 到 Ring ReduceScatter CommPlan
  3. FluxLocalCompute 绑定 Compute 域 LirOp（不独立 lowering）
  4. FluxAllGather lowering 到 Ring AllGather CommPlan
  5. 三阶段 DAG 依赖边表达：FluxReduceScatter → FluxLocalCompute → FluxAllGather
  6. FLUX 仅在 OverlapHint::ForceFlux 时激活
  7. world_size ≥ 2 前提条件

REQ-LIR-022: Algorithm Selection IR 表达

LIR 表达通信算法选择语义。通信算法选择是 LIR Communication Domain 的核心决策点。算法选择基于消息大小、拓扑、延迟/带宽偏好。LIR 表达选择的算法，gllm-nccl 执行对应的 lower 路径。

  • 算法枚举：Ring / Tree / TopoAwareRing / CollNetDirect / ChunkedPipeline / HierarchicalRing / HardwareReduce / FluxPipeline
验收标准：
  1. CommAlgorithmSelect 定义为 Communication 域 LirOp 变体
  2. 8 种算法枚举完整定义
  3. Ring → ring_lower 映射正确
  4. Tree → tree_lower 映射正确
  5. FluxPipeline → flux_lower 映射正确

REQ-LIR-023: Prefix Sharing IR 表达

LIR 表达分布式场景下 KV 前缀共享语义。RadixAttention 用 Radix Tree 匹配共享前缀，命中时复用已有 KV cache 避免重复计算。LIR 表达前缀匹配、复用、淘汰的语义。

  • PrefixReuse{prefix_len, kv_offset} — 前缀 KV 复用绑定
  • PrefixEvict{policy, min_evict_len} — 前缀 KV 淘汰
验收标准：
  1. PrefixMatch/PrefixReuse/PrefixEvict 定义为 Communication 域 LirOp 变体
  2. PrefixMatch lowering 到 KvPrefixIndex 查询
  3. PrefixReuse lowering 到 KV cache offset 绑定
  4. PrefixEvict lowering 到 KV cache 淘汰策略

  6. 分布式场景需跨 rank 的 KV 页迁移

REQ-LIR-024: Chunked Prefill IR 表达

LIR 表达长序列 prefill 分块执行语义。长序列 prefill 分块执行，避免单次 prefill 占用全部计算资源。每个 chunk 独立执行 attention，chunk 间通过 KV cache 传递状态。分布式场景下 chunk 调度与通信重叠。

来源：SGLang ✅ / vLLM ✅ / gllm ✅（scheduler/chunked_prefill.rs）

核心 IR 元素：

  • ChunkedPrefill{chunk_size, total_seq_len, chunk_idx} — 分块 prefill 执行
  • ChunkBoundary{is_first, is_last} — chunk 边界标记
验收标准：
  1. ChunkedPrefill/ChunkBoundary 定义为 Communication 域 LirOp 变体
  2. ChunkedPrefill lowering 到 SymDim::Symbolic("seq_len") 绑定 + Prefill/Decode 路径切换
  3. ChunkBoundary lowering 到 KV cache 追加模式
  4. chunk_size 由调度器动态决定（非编译时常量）
  5. 最后一个 chunk 可能小于 chunk_size（SymDim 正确处理）

REQ-LIR-025: Speculative Decoding Distributed IR 表达

LIR 表达分布式推测解码的 draft-verify 通信语义。Draft 模型在独立 rank 或低优先级 SM 分区生成候选 token，Verify 阶段在目标模型验证。DraftKvSync 表达 draft 模型 KV cache 与目标模型的同步。

来源：SGLang ✅（DFlash/EAGLE）/ vLLM ✅（EAGLE）/ gllm ✅（speculative/）

核心 IR 元素：

  • DraftGenerate{num_candidates, draft_model_id} — Draft 模型候选生成
  • VerifyAccept{candidate_tokens, accept_mask} — 目标模型验证接受
  • DraftKvSync{layer_range, mode} — Draft-Target KV cache 同步（mode: FullSync/Incremental）
验收标准：
  1. DraftGenerate/VerifyAccept/DraftKvSync 定义为 Communication 域 LirOp 变体
  2. DraftGenerate lowering 到小模型 CompilerGraph 编译路径
  3. VerifyAccept lowering 到目标模型 forward + token 比对
  4. DraftKvSync lowering 到 KV cache 页迁移（复用 KvTransferOut/In）
  5. 单机场景下 Draft/Verify 共享 GPU 通过 SM 分区隔离
  6. 分布式场景通过 P2P 传输

REQ-LIR-026: EPLB (Expert Placement Load Balancing) IR 表达

LIR 表达 EPLB 动态重分配专家权重到不同 rank 的语义。基于历史 token 负载均衡，算法包括 deepseek（贪心）、deepseek_vec（向量化）、elasticity_aware（弹性感知）。LIR 表达负载记录、重均衡决策、权重迁移。

来源：SGLang ✅（EPLB with deepseek/deepseek_vec/elasticity_aware 算法）/ vLLM ❌ / gllm ❌（预留 ExpertPlacement）

核心 IR 元素：

  • ExpertLoadRecord{num_layers, num_physical_experts} — 专家负载记录
  • ExpertRebalance{algorithm, num_replicas, num_groups} — 专家重均衡决策（algorithm: Deepseek/DeepseekVec/ElasticityAware）
  • ExpertWeightMigrate{src_rank, dst_rank, expert_indices} — 专家权重迁移
验收标准：
  1. ExpertLoadRecord/ExpertRebalance/ExpertWeightMigrate 定义为 Communication 域 LirOp 变体
  2. ExpertLoadRecord lowering 到 MoE dispatch 时的负载累加
  3. ExpertRebalance lowering 到 AllReduce 负载 + 贪心/向量化算法
  4. ExpertWeightMigrate lowering 到 P2P 权重传输 + 映射表更新
  5. EPLB 仅在 ExpertPlacement::HotCold 时激活
  6. 重均衡频率由 eplb_config.step_interval 控制

14. Communication Domain Decision 定义

DEC-LIR-009: Communication Domain 作为第四域（vs 子域 vs 独立 pass）

Communication 语义在 LIR 域划分中的位置：独立第四域 vs Compute 域子域 vs 独立 pass。

方案选项

  • Option A: Compute 域子域（通信归入计算，语义耦合，pass 调度不灵活）
  • Option B: 独立 pass 在 Compute 域执行（通信 pass 与计算 pass 混排，依赖隐式）
  • Option C: 独立第四域 Communication（chosen — 通信语义正交于计算/内存/布局，独立域使 pass 更聚焦，nccl feature-gated 零开销）

选定方案

Option C：独立第四域 Communication

理由

通信语义（AllReduce/AllToAll/P2P）与计算语义正交：通信操作不执行算术计算，而是跨 rank 数据移动与同步。独立域使通信融合/调度/优化 pass 聚焦，避免与 Compute 域融合 pass 混排导致调度冲突。nccl feature-gated 保证非分布式场景零开销。

DEC-LIR-010: PP2 微批次粒度选择

Pipeline Parallel 2-stage 的微批次切分粒度：token-level vs request-level vs layer-level。

方案选项

  • Option A: token-level 微批次（粒度最细，通信最频繁，调度开销大）
  • Option B: request-level 微批次（chosen — 按请求切分，通信频率适中，调度开销可控）
  • Option C: layer-level 微批次（粒度太粗，2-stage 流水线气泡率过高）

选定方案

Option B：request-level 微批次

理由

2-stage 流水线中 request-level 微批次平衡了通信频率与流水线气泡率。每个 micro-batch 包含若干请求的完整 forward pass，stage 间通信一次激活传输。token-level 粒度导致 stage 间通信过于频繁，调度开销占比过高。layer-level 粒度太粗，2-stage 下流水线气泡率接近 50%。

DEC-LIR-011: MC2 chunk 分割策略

Communication-Compute Overlap 的 chunk 分割策略：等分 vs 自适应。

方案选项

  • Option A: 等分 chunk（实现简单，但通信-计算时间不匹配时效率低）
  • Option B: 自适应 chunk（chosen — 按通信/计算时间比动态分割，重叠率最优）

选定方案

Option B：自适应 chunk

理由

自适应 chunk 根据 AllReduce 通信时间与后续计算时间的比值动态分割 chunk 大小，使通信流处理 chunk N 时计算流恰好完成 chunk N-1 的计算，最大化重叠率。等分 chunk 在通信/计算时间不匹配时产生等待间隙。自适应策略由 CommSchedulePass 根据 DeviceProfile（NVLink 带宽/SM 数量）和 ComputePattern 计算量估算推导。

DEC-LIR-012: UNIEP dispatch/combine 融合 vs 分离

UNIEP AllToAll dispatch 与 combine 的 IR 表达：融合为单一 LirOp vs 分离为两个独立 LirOp。

方案选项

  • Option A: 融合为单一 AllToAll LirOp（dispatch+combine 封装，但中间状态不可优化）
  • Option B: 分离为 AllToAllDispatch + AllToAllCombine 两个 LirOp（chosen — 中间 expert 计算可插入优化 pass）

选定方案

Option B：分离为 AllToAllDispatch + AllToAllCombine

理由

dispatch 与 combine 之间存在 expert FFN 计算，分离为两个 LirOp 使 expert 计算可作为 Compute 域 LirOp 插入 dispatch/combine 之间，接受 Compute 域融合 pass 优化。融合为单一 LirOp 将 expert 计算封装在内部，Compute 域 pass 无法触及。分离后 dispatch→expert_compute→combine 的三段结构清晰，对应 MoE 路由→计算→收集的物理数据流。

DEC-LIR-013: TP 切分策略选择

权重切分按行（Row）vs 按列（Col）的选择影响通信模式。Row 切分输出需 AllReduce；Col 切分输出需 ReduceScatter+AllGather。选择依据：GEMM 形状、通信量、硬件拓扑。

  • 选项 A：Row 切分 — AllReduce 通信量 = hidden_size × dtype_size，适合 hidden_size 较小
  • 选项 B：Col 切分 — ReduceScatter+AllGather 通信量 = output_size × dtype_size / world_size，适合 output_size 较大

决策：LIR 不硬编码选择，由 CommSchedulePass 根据 DeviceProfile + 消息大小自动选择。权重切分策略在 BUILD 阶段由 CompilerGraph 确定。

DEC-LIR-014: 通信量化策略选择

通信量化（FP32→INT8/FP8）节省 4x 带宽，但引入精度损失。量化策略由 CommCompressHint 驱动。

  • 选项 A：AlwaysCompress — 所有 AllReduce 通信前量化，带宽受限场景最优
  • 选项 B：NeverCompress — 不量化，精度优先场景
  • 选项 C：Auto — 基于消息大小自适应：小消息不量化（延迟主导），大消息量化（带宽主导）
  • 选项 D：ForceQuant(scheme) — 用户指定量化方案

决策：Auto 为默认。CommOptimizePass 根据 CommCompressHint + 消息大小 + 硬件带宽自动决策。量化精度损失由 SPEC 10-QUALITY 约束（相对误差 < 1e-3）。

DEC-LIR-015: PD 分离传输协议选择

PD 分离架构中 KV 迁移传输协议选择：NVLink（同节点，带宽 ~300 GB/s）vs RDMA（跨节点，带宽 ~50 GB/s）。

  • 选项 A：NVLink — 同节点，延迟低带宽高，KvTransferOut/In lowering 到 P2pCopy
  • 选项 B：RDMA — 跨节点，需 RDMA 注册+传输，lowering 到 RdmaPageFetch/RdmaPagePush

决策：由 SystemTopology 自动探测。PdDisaggConfig.role=Auto 时自动选择；同节点走 NVLink，跨节点走 RDMA。LIR 层不硬编码，CommSchedulePass 根据 topology 输出决定。

DEC-LIR-016: EPLB 算法选择

EPLB 专家负载均衡算法选择：deepseek（贪心）/ deepseek_vec（向量化）/ elasticity_aware（弹性感知）。

  • 选项 A：Deepseek — 贪心算法，O(E×R) 复杂度，适合专家数较少
  • 选项 B：DeepseekVec — 向量化贪心，O(E×R/lanes)，SIMD 加速，适合专家数较多
  • 选项 C：ElasticityAware — 弹性感知，考虑专家复制+迁移成本，适合高动态负载

决策：由 ExpertPlacement 驱动。RoundRobin 时不需要 EPLB；HotCold 时默认 DeepseekVec（平衡性能和精度）。用户可通过 MoeDistributedConfig 覆盖。

DEC-LIR-017: FLUX vs MC2 通信重叠策略

FLUX（ReduceScatter→LocalCompute→AllGather）vs MC2（AllReduceChunk 双流交替）通信-计算重叠策略选择。

  • 选项 A：MC2 — AllReduceChunk chunk 级交替，实现简单，SM80+ 可达 60-80% 重叠
  • 选项 B：FLUX — ReduceScatter→LocalCompute→AllGather 三阶段，理论 ≥90% 重叠，但需要 ReduceScatter 支持

决策：OverlapHint 驱动。PreferOverlap/ForceDoubleBuffer → MC2；ForceFlux → FLUX。Auto 时 CommSchedulePass 根据硬件能力自动选择（SM90+ 优先 FLUX，SM80-89 优先 MC2）。

DEC-LIR-018: CP Ring Attention 与 TP 的兼容

Context Parallelism 与 Tensor Parallelism 共存时，Ring Attention 的 KV chunk 传递需要在 TP group 内同步。

  • 选项 A：CP-TP 分离 — CP 组独立于 TP 组，Ring Attention 在 CP 组内传递 KV，TP 组内正常 AllReduce
  • 选项 B：CP-TP 融合 — Ring Attention 同时处理 CP 和 TP 通信，减少同步次数

决策：选项 A（CP-TP 分离）。CP group 和 TP group 是独立的通信域，RingAttention 在 CP group 内传递 KV，AllReduceSync 在 TP group 内聚合。通信域正交，零干扰。

15. Communication Domain Dataflow 定义

DF-LIR-005: AllReduceChunk lowering

Communication 域 AllReduce 语义 LirOp lowering 到 AllReduceChunk VmInstr。AllReduceChunk 携带 sendbuf/recvbuf/count/dtype/op/rank/world_size/chunk_idx，直接映射到 VmInstr::AllReduceChunk。

契约

  • from: CommComputeOverlap LirOp (Communication 域)
  • to: VmInstr::AllReduceChunk 序列
  • transform: comm_lir_to_vminstr(CommComputeOverlap) → Vec<VmInstr::AllReduceChunk>
  • direction: sync（编译时一次性）
  • feature: nccl

DF-LIR-006: Pipeline Stage 激活传递

PP2 流水线 stage 间激活数据传递：StageSend(stage_0) → inter-node 传输 → StageRecv(stage_1)。激活数据通过 P2pPageFetch（同节点 NVLink）或 RdmaPageFetch（跨节点 RDMA）传输。

契约

  • from: StageSend LirOp (stage_id=0)
  • to: StageRecv LirOp (stage_id=1)
  • transform: StageSend → P2pPageFetch/RdmaPageFetch → StageRecv
  • direction: async（运行时异步传输）
  • feature: nccl

DF-LIR-007: Expert AllToAll dispatch/combine 数据流

UNIEP 专家并行 AllToAll 数据流：AllToAllDispatch 按专家路由表将 token 分发到目标 rank → 目标 rank 执行 expert FFN 计算 → AllToAllCombine 按源 rank 收集 expert 输出合并回原 token 顺序。

契约

  • from: AllToAllDispatch LirOp (本 rank tokens)
  • to: AllToAllCombine LirOp (本 rank combined outputs)
  • transform: dispatch(token→expert_rank) → expert_compute → combine(expert_rank→token)
  • direction: async（运行时 AllToAll 异步通信）
  • feature: nccl

DF-LIR-008: Comm-Compute overlap chunk 调度数据流

MC2 通信-计算重叠的 chunk 调度数据流：CommComputeOverlap LirOp 根据 OverlapHint 和 ChunkPartition 生成 AllReduceChunk + DualStreamBarrier VmInstr 序列。ForceDoubleBuffer 模式生成双缓冲交替序列；ForceFlux 模式生成流式 chunk 序列；PreferIsolated 模式生成顺序执行序列。

契约

  • from: CommComputeOverlap LirOp + OverlapHint + ChunkPartition
  • to: Vec<VmInstr> (AllReduceChunk + DualStreamBarrier 交替序列)
  • transform: overlap_schedule(CommComputeOverlap, OverlapHint) → Vec<VmInstr>
  • direction: sync（编译时调度生成）
  • feature: nccl

DF-LIR-009: TP 权重切分到 AllReduce 聚合

TP 权重切分数据流：WeightShard(axis=Row) → 每个 rank 独立计算 GEMV → AllReduceSync(Sum) 聚合部分结果。WeightShard(axis=Col) → 每个 rank 独立计算 GEMV → ReduceScatterSlice + AllGather 聚合。

契约

  • from: WeightShard LirOp (Communication 域)
  • to: AllReduceSync/ReduceScatterSlice LirOp (Communication 域)
  • transform: tp_shard_weight(weight, rank, axis) → partial_result → allreduce/partial_result → complete_result
  • direction: sync（编译时确定切分+聚合模式）
  • feature: nccl

DF-LIR-010: PD 分离 KV 迁移数据流

PD 分离 KV 迁移数据流：KvTransferOut（Prefill 节点）→ RDMA/NVLink 传输 → KvTransferIn（Decode 节点）。传输以 page 为粒度，PdHandshake 保证两端同步。

契约

  • from: KvTransferOut LirOp (Prefill rank)
  • to: KvTransferIn LirOp (Decode rank)
  • transform: kv_pages → rdma/nvlink_transfer → kv_pages_bound
  • direction: async（运行时异步传输）
  • feature: nccl

DF-LIR-011: 通信量化数据流

通信量化数据流：CommQuantize(FP32→INT8/FP8) → AllReduceChunk(低精度通信) → CommDequantize(INT8/FP8→FP32)。Delta Encoding 路径：CommDeltaEncode → CommBitPack → 通信 → CommBitUnpack → CommDeltaDecode。

契约

  • from: CommQuantize LirOp (Communication 域)
  • to: CommDequantize LirOp (Communication 域)
  • transform: fp32_data → quantize → low_precision_allreduce → dequantize → fp32_data
  • direction: sync（编译时确定量化策略）
  • feature: nccl

DF-LIR-012: FLUX 三阶段数据流

FLUX 分解数据流：FluxReduceScatter → 每个 rank 持有 home chunk → FluxLocalCompute（与 FluxAllGather 并行）→ FluxAllGather 收集所有 chunk。三阶段 DAG 依赖：FluxReduceScatter → FluxLocalCompute → FluxAllGather。

契约

  • from: FluxReduceScatter LirOp
  • to: FluxAllGather LirOp
  • transform: all_data → reduce_scatter → home_chunks → local_compute → partial_results → all_gather → complete_results
  • direction: async（运行时异步重叠）
  • feature: nccl

16. Communication Domain Controlflow 定义

CF-LIR-003: 通信域 pass pipeline 执行顺序

Communication 域 pass pipeline 按固定顺序执行：CommFusionPass → CommSchedulePass → CommOptimizePass。CommFusionPass 先合并冗余通信（减少通信量）；CommSchedulePass 次之（根据 OverlapHint 调度 chunk 级重叠）；CommOptimizePass 最后（量化压缩进一步减少通信量）。Communication 域在 Memory 域之后执行，依赖 Compute 和 Memory 域结果确定通信模式和缓冲区布局。

CF-LIR-004: Pipeline 1F1B 调度控制流

PP2 1F1B（One Forward One Backward）调度控制流：stage 0 执行 forward micro-batch 0 → StageSend → stage 1 执行 forward micro-batch 0 → stage 0 执行 forward micro-batch 1（与 stage 1 的 forward micro-batch 0 重叠）→ ... → 稳态阶段每个 stage 同时执行 forward N+1 和 backward N-1。StageBoundary 同步点保证 stage 间激活传递的顺序性。

CF-LIR-005: TP 切分→计算→聚合执行顺序

TP 执行顺序：WeightShard 加载切分权重 → Compute 域执行 GEMV/GEMM → AllReduceSync/ReduceScatterSlice 聚合。切分和聚合由 Communication 域 pass pipeline 保证顺序。WeightShard 在 BUILD 阶段确定，AllReduceSync 在 CommSchedulePass 中调度。

CF-LIR-006: FLUX 三阶段执行控制流

FLUX 三阶段严格顺序：FluxReduceScatter 完成前 FluxLocalCompute 不启动；FluxLocalCompute 在 home chunk 上执行计算；FluxAllGather 与 FluxLocalCompute 可并行（home chunk 计算完成后立即启动对应 AllGather step）。CommSchedulePass 生成三阶段 VmInstr 序列 + DualStreamBarrier 同步。
