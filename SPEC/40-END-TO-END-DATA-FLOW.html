端到端数据流 (SSOT)

> 核心定位: 本文档是用户 API 调用到推理结果返回的完整数据流唯一真源。从 client.generate(prompt) 到 GenerationResponse 的每一步数据转换，在此文档中有且仅有一个权威描述。任何代码注释、变量名或口头描述与此文档矛盾时，以此文档为准。

> 与 SPEC 39 的关系: SPEC 39 定义编译器行为（编译时），本文档覆盖编译时 + 推理时全链路。SPEC 39 的 §0.1 两仓职责边界在本文档中有完整的端到端数据流验证。

> 与 ARCH-DATA-FLOW-CONTRACT 的关系: ARCH-DATA-FLOW-CONTRACT 定义 executor/loader 内部数据流的唯一来源契约，本文档覆盖更广范围（从用户 API 到结果返回），但 executor/loader 内部数据流的 SSOT 仍以 ARCH-DATA-FLOW-CONTRACT 为准。

> 三时态重组 (A5-1 修复): 原文档混合了 BUILD-TIME、COMPILE-TIME、RUNTIME 三时态描述。本次修复按三时态重组为三个 Part，每个 Part 头部明确标注时态标签，读者可清晰识别约束生效时机。

§4.3 OutputMode 裁剪

OutputMode 决定图的尾部 ops（executor_compile.rs:483-497）：

ModelKindOutputMode图尾部 ops
EmbeddingEncodeToLayer { anchor_layer, pool_mode }MeanPool + L2Norm
RerankerClassifyBinary { positive/negative_token_id }CLS pooling + ClassifierDense
ClassifierClassifyMultiway { label_token_ids }ClassifierDense + ClassifierOutProj

OutputMode 只裁剪尾部 ops，不修改模型大类。 模型大类由 ArchitectureFeatures.family 决定。OutputMode 被放入 BusinessConfig.output_modes，图构建器根据此决定尾部 ops。裁剪后的 CompilerGraph 编码了所有编译信息，编译器零 OutputMode 访问。

§13.2.1 结构差异（Family 驱动 BUILD 侧裁剪）

图构建器（build_graph.inc.rs）不按 OutputMode 物理裁剪 ops。图始终构建完整的模型结构。裁剪通过两种机制实现：

结构差异（Family 驱动）：Encoder（post-norm）和 Decoder（pre-norm）的图结构本身不同，这是 ArchitectureFeatures.family 驱动的，不是 OutputMode 驱动的：

Family图尾部 ops条件
Encoder (post-norm)MeanPool → (可选 classifier head: Dense → Tanh → OutProj)features.is_post_norm = true
Decoder (pre-norm)final_norm → lm_head → (LogitSoftcap) → (MTP) → Argmax → StoreToken → CheckStopConditionfeatures.is_post_norm = false

这是 BUILD 阶段的结构决策。不同的 Family 产生根本不同的图拓扑，编译器后续从图拓扑推导控制流（见 Part B §13.2.2）。

§11 Session KV Cache 复用

§11.1 session_position 参数传递

session_position 通过 ABI arg 16 传入 JIT 机器码，控制 KV cache 复用行为：

  • session_position = 0：全新请求，从 position 0 开始 prefill
  • session_position > 0：resume 模式，跳过已有 KV cache 的 prefill，从 session_position 继续

§11.2 Resume 模式数据流

client.generate("续写") with session_position = N
  → input_ids = [续写 token ids]
  → positions  = [N, N+1, ..., N+len-1]
  → JIT 内部: 跳过前 N 个 position 的 KV 写入
              从 position N 开始计算 attention
              新 KV cache 追加到已有 cache 之后

§11.3 Position-agnostic RoPE

对系统前缀页设置 cos=1, sin=0（REQ-KV-OPT-010），确保 prefix KV cache 与具体 position 无关，可在不同请求间安全复用。

build_business_config()
  → build_output_modes(manifest.kind, max_new_tokens, eos_id)
     → Vec<OutputMode>  // Embedding→EncodeToLayer, Reranker→ClassifyBinary, Chat→Generate
→ build_compiler_graph(features, config, ..., business_config, ...)
  → 图构建器根据 output_modes 决定尾部 ops
  → 产出不同裁剪的 CompilerGraph
→ compile_from_auto_graph(graph, ...)
  → 不同图 → 不同 JIT 机器码（编译时单态化）

关键：OutputMode 裁剪发生在编译时，运行时执行的是已经裁剪好的 JIT 机器码。不存在运行时的 OutputMode 分支判断。

REQ-E2E-001 §0 两阶段模型与 BUILD/COMPILE 边界

端到端数据流必须遵循两阶段模型：编译时一次性 + 推理时每次调用。编译时内部存在 BUILD 和 COMPILE 两个正交阶段，由 build_compiler_graph() 返回值作为分界线。

验收标准：
  • 推理热路径零编译行为：执行 grep -rn 'compile\|build_compiler_graph' src/engine/mega_kernel/executor_core.inc.rs src/engine/mega_kernel/executor_ops.inc.rs 返回 0 匹配。
  • BUILD 阶段产物可识别：build_compiler_graph() 返回 CompilerGraph 作为 BUILD→COMPILE 边界标志，调用点位于 src/arch/auto_graph.rs，禁止出现在 src/engine/ 或 src/jit/。
  • COMPILE 唯一入口：InferenceCompiler::compile()（API POST /compile）是 MegaKernelFn 的唯一生产者，grep -rn 'fn compile\b\|fn compile(' src/ 仅命中此入口。
  • 推理时一次 CALL：generate_single_sequence() 通过 MegaKernelFn ABI（unsafe extern "C" fn(ctx: *const u8) -> u32）完成完整 mega-kernel，单次函数指针调用，运行时无再编译。
  • E2E 测试断言（cargo test --test test_e2e_generator）：单次 MegaKernelFn 调用后 output_buffer 中存在完整 UTF-8 文本，验证运行时零编译。

  • 双模型 ABI 一致：reranker 与 generator 共享相同 MegaKernelFn ABI（unsafe extern "C" fn(ctx: *const u8) -> u32），仅 KernelContext 内容不同。
  • E2E 测试断言：cargo test --test test_e2e_reranker_then_generator 通过，reranker 输出 top-k 后 generator 基于此生成，整链端到端无 panic。
