> 核心定位: gllm-kernels 是一个通用编译器，不是模型推理框架。编译器不假设图的结构——喂什么编译什么。1 个 op 的 Gather 图和 35 层 decoder 图走同一管线，产出同一 ABI 的 JIT 机器码。

§0 设计宪法

编译器 = 喂什么编译什么。

这条原则贯穿所有设计决策。任何违反此原则的现状（代码或 SPEC）都是 bug，必须修正。

推论：

  • 编译器不知道"encoder"和"decoder"的区别——它只看到一个 CompilerGraph
  • 编译器不知道"embed 阶段"和"lm_head 阶段"——它只看到 ops 的拓扑排序
  • 编译器不知道"采样循环"——它只看到图中有没有 Argmax/StoreToken/CheckStopCondition
  • 编译器不知道"权重布局是 embed→layers→lm_head"——它只看到 named_offsets 映射表
  • 模型形态差异 100% 由 CompilerGraph 拓扑编码，编译器零假设

§0.1 两仓职责边界与参数硬化分类

> 核心分界线: gllm 拥有"服务形态决策"，gllm-kernels 拥有"图拓扑推导"。JIT 编译产物的每一条指令，要么在 compile() 时确定（编译时硬化），要么在 MegaKernelFn() 时传入（运行时参数）。不存在第三种时态。

§0.1.1 两仓职责边界

gllm（推理服务器）和 gllm-kernels（JIT 编译器）是两个正交的职责域。混淆两者是 SPEC 39 重构前最主要的架构违反源头。

职责gllm（推理服务器）gllm-kernels（JIT 编译器）
决定用户可用的服务形态✅ OutputMode 列表（Generate/Classify/Encode）❌ 不知道 OutputMode 的存在
根据模型架构 + 服务形态构建 CompilerGraph✅ 构建"含采样管线"或"不含采样管线"的图❌ 不关心"为什么"要编译这个图
从图拓扑推导代码生成策略❌ 不需要告诉编译器"这是 encoder"✅ 图有 Argmax → emit 采样管线；无 → 不 emit
编译时硬化哪些尾部路径间接决定（通过图的 op 组成）✅ 图有对应 op → emit 路径代码
运行时路由用户请求到具体路径✅ 选择调用哪个 MegaKernel✅ 直接执行裁剪后的唯一路径
两个正交维度：
维度含义决定者何时确定
图拓扑有没有 Argmax/StoreToken/CheckStopConditiongllm 构建 CompilerGraph 时编译时（图的内容）
服务形态Generate/Classify/Encode用户请求运行时（选择路径）

图拓扑决定了服务形态的上界：无 Argmax 的图没有采样管线 → 只能 Classify/Encode；有 Argmax 的图有采样管线 → 可 Generate/Classify/Encode。但编译器不判断"这是什么类型的图"——它只检查图中存在哪些 OpKind。

禁止的职责越界：
  • X gllm 传递 is_encoder: bool 给 gllm-kernels — 编译器从图拓扑推导，不需要外部告知
  • X gllm-kernels 从 output_modes 反推 is_encoder — OutputMode 是服务形态概念，不是图结构概念
  • X gllm-kernels 用 is_encoder 控制代码生成分支 — 应从 graph.contains_op_kind(OpKind::Argmax) 等拓扑查询推导
  • X gllm-kernels 知道"encoder"或"decoder"的含义 — 它只看到 CompilerGraph

§0.1.2 编译时硬化（bake 进 JIT 机器码）

以下信息在 compile(graph, config) 调用时确定，直接硬编码为 JIT 机器码中的指令序列或立即数。编译后不可变更，变更需重新编译。

编译时硬化分为两类：图拓扑推导属性（编译器自主推导，不接受外部参数）和外部编译配置（调用方通过 CompileConfig 传入）。

A. 图拓扑推导属性（编译器自主推导）

以下属性由 gllm-kernels 从 CompilerGraph 拓扑直接推导，不接受外部参数覆盖。任何试图通过 CompileConfig 或函数参数传入这些值的行为都是架构违反。

§0.1.4 判定规则

给定任意一条 JIT 产物中的信息，判定其属于编译时还是运行时：

  1. 该信息在编译时已知吗？→ 已知 → 编译时硬化（例：vocab_size 从 OpKind::Argmax { vocab_size } 推导，编译时确定）
  2. 该信息在每次推理调用间会变化吗？→ 会变化 → 运行时参数（例：temperature 每次调用可能不同）
  3. 该信息的不同值需要不同的机器码吗？→ 需要 → 编译时硬化（例：BF16 vs F32 需要不同指令）；不需要 → 运行时参数（例：temperature 值不同但使用同一套采样指令）
  4. 该信息能从图拓扑推导吗？→ 能 → 图拓扑推导属性（禁止外部参数传入）；不能 → 外部编译配置或运行时参数
关键推论：
  • 采样管线代码的存在性是图拓扑推导（图有 Argmax → emit；无 → 不 emit）
  • 采样管线参数的值是运行时传入（temperature/top_k/top_p 从 ABI 参数读取）
  • OutputMode 在 BUILD 阶段决定图裁剪策略，编译时写死唯一路径。裁剪后的图 ops 编码了所有编译信息
  • 循环 bound 的类型是图拓扑推导（图有 Argmax → DynamicVReg；无 → Const(1)）；DynamicVReg 对应的实际值从 ABI 的 max_new_tokens + prompt_len 运行时计算
  • kv_cache_ptr 的加载指令存在性是图拓扑推导（图有 KvCacheWrite op → 加载；无 → 不加载）；指针值是运行时 ABI 传入

REQ-UMK-011 CompileConfig 定义 (REQ-UMK-011)

定义 CompileConfig 结构体，仅携带编译器所需的非图推导参数。BusinessConfig 已彻底分离——编译器不持有 BusinessConfig 引用，结构上保证编译器无法访问 output_modes 等业务字段。debug_jit 是编译器唯一直接读取的非图推导参数，因为它影响全局 codegen 行为（INT3 插入、source map 生成），无法通过 OpKind 表达。

定义 CompileConfig 结构体，仅携带编译器所需的非图推导参数（内存上界、权重偏移映射、采样配置、设备能力等）。禁止包含任何图结构假设字段（is_encoder/inference_mode/vocab_size/num_layers/hidden），这些信息全部由 CompilerGraph 拓扑自动推导。字段来源严格遵循 ARCH-DATA-FLOW-CONTRACT，每个字段都有唯一合法来源。

/// 编译配置 -- 携带编译器需要的非图推导参数。
///
/// 设计原则：CompileConfig 不携带任何图结构假设。
/// 编译器从 CompilerGraph 推导一切结构信息。
/// CompileConfig 只提供"图本身无法推导的外部输入"。
///
/// BusinessConfig 已彻底分离：编译器不持有 BusinessConfig 引用，
/// 结构上保证编译器无法访问 output_modes 等业务字段。
/// BusinessConfig 仅在 BUILD 阶段（图构建）使用，不传入编译器。
pub struct CompileConfig {
    // -- 内存上界 --
    /// Buffer 分配上界（替代 SYMDIM_MAX_SEQ_LEN 硬编码）。
    /// 来自模型配置 max_position_embeddings，不是编译器硬编码。
    /// 仅用于 scratchpad/KV buffer 分配大小，不影响循环 bound。
    pub max_seq_len: usize,
    // -- 调试 --
    /// 启用 JIT 调试输出（INT3 breakpoints + source map）。
    /// 这是编译器唯一直接读取的非图推导参数——debug_jit 影响全局
    /// codegen 行为（INT3 插入、source map 生成），无法通过 OpKind 表达。
    pub debug_jit: bool,
    // -- 异构层配置 --
    /// 异构层布局（Gemma 4 E2B 等）。None = 均匀层。
    /// 编译器通过 GprCondAction 在 JIT 内条件分支，不由 CompileConfig 驱动。
    pub hetero: Option<HeteroLayerConfig>,
}

§1.3.3 ForwardPhaseDispatch 详细行为 (REQ-UMK-013) ForwardPhaseDispatch 是 JIT 内的三路分支，由 phase_id 寄存器控制：

§3 权重布局

§3.1 通用权重布局：named_offsets 映射表

编译器不假设权重是"embed→layers→lm_head"布局。权重布局由 CompilerGraph.weight_layout() 返回的 named_offsets 映射表描述：

named_offsets: HashMap<String, usize>
  "embed"     -> 0
  "layer.0"   -> 262144
  "layer.1"   -> 524288
  ...
  "lm_head"   -> 8388608

JIT 机器码通过 weight_ptr + named_offsets["layer.0"] 访问权重。偏移值 bake 为立即数。

对于小图：named_offsets 只包含小图需要的权重条目（如 "sg_embed_table" -> 0）。

§3.2 模型级权重布局 (MegaKernelWeightLayout) (REQ-UMK-008)

embed_offset -> layer_0_offset -> [layer_stride x N] -> final_norm_offset -> lm_head_offset

这是pack 阶段的便利结构，不是编译器的假设。编译器只读 named_offsets，不读 MegaKernelWeightLayout。

§7 编译管线三层并行化

核心洞察: mega-kernel 编译完全串行处理所有 fusion groups。异构模型（Gemma-4）有 4 种层类型（ss/fs/sl/fl），每种类型有独立的 fusion group 序列，它们之间编译完全独立。三层并行化将编译时间从 O(总 groups) 降低到 O(最慢层类型)。

§7.1 基础设施：LoweringContext（REQ-UMK-015）

问题: emit_fusion_groups 有 16 个参数，大量参数在调用链中重复传递。emit_standalone_op、emit_gemm_inline_with_hook 等函数同样参数膨胀。新增 ResourceBudget 等参数会进一步恶化。

方案: LoweringContext<'a> 持有编译会话的所有不可变共享状态：SimdWidth、SymDimSlotMap、IsaHook、ResourceBudget、RopeCacheRequirement、PleScratchRequirement、DwcScratchRequirement。

约束: LoweringContext 一旦构造就不可变 — 编译会话期间 width/sym_map/hook/budget 不变。通过 &LoweringContext 传递，不消耗所有权。所有 emit 函数签名简化为 ctx: &LoweringContext + 必要的局部参数。

§7.2 EmitState — 可变状态抽取（REQ-UMK-016）

问题: emit_fusion_groups 内部有 5 个跨迭代可变状态变量：current_abi（ABI 指针集）、hetero_phase（异构层模板阶段追踪）、in_layer_loop（是否已进入层循环）、hetero_seg_byte_offset（异构段偏移）、hetero_seg_weight_base（异构段权重基址）。这些状态变量阻止了 emit_fusion_groups 被拆分为可并行的子函数。

方案: 抽取为 EmitState 结构体。

状态机模型: emit_fusion_groups 的执行是一个有限状态机：

  • HeteroPhase: BeforeLayers（层循环前融合组）→ SlidingSmall(ss) → FullSmall(fs) → SlidingLarge(sl) → FullLarge(fl)
  • in_layer_loop: false（层循环外）→ true（层循环内）

状态转移: BeforeLayers 遇到 layer_sliding_small op → SlidingSmall(0); SlidingSmall(i) 完成 → SlidingSmall(i+1) 或 FullSmall; FullSmall 遇到 layer_sliding_large op → SlidingLarge(0); SlidingLarge(i) 完成 → SlidingLarge(i+1) 或 FullLarge; FullLarge 遇到非 layer op → BeforeLayers（层循环后融合组）。

并行化关键: 每种层类型（ss/fs/sl/fl）的编译拥有独立的 EmitState 快照。层模板编译函数 compile_layer_type_body 从快照初始化自己的 EmitState，编译完成后返回 LayerTemplate。

§7.3 第一层：层模板并行编译（最大收益, REQ-UMK-017）

核心洞察: 异构模型（Gemma-4）的 4 种层类型各自有一组 fusion groups。4 种类型之间编译完全独立，可以用 rayon 并行编译。

数据结构: LayerTemplate 封装编译后的 VmProgram + LayerAbiMap（input_ptr/weight_ptr/output_ptr/scratch_base 4 个 VRegId 映射）。

编译函数: compile_layer_type_body(ctx, group_range, plan, graph, alloc, state, resolver) → Result<LayerTemplate>

并行实例化: 在 emit_fusion_groups 中：

  1. 预分析 FusionPlan，识别每种层类型的 group 边界（分析 anchor op label 前缀）
  2. 使用 rayon 并行编译 N 个模板：layer_types.par_iter().map(compile_layer_type_body).collect()
  3. 在主 VmProgram 中按顺序实例化模板：prog.append_with_mapping(template, &abi_subst)

VmProgram 模板合并: append_with_mapping 实现替换语义：模板内 DeclareVReg → 跳过声明用 subst 映射替代；模板内 VRegId 引用 → 全部替换为主程序对应的 VRegId；LoopBegin/LoopEnd → 保留循环结构嵌入；新分配的 VReg → 主程序分配新 VRegId。

FusionMode 模板编译可行性: Standalone/LoopFusion ✅ 已支持直接 emit; EpilogueInjection/NormIntoGemm/QkvSharedInput/FFNBlock/FusedQkvNormRope 可行整体编译; TileLevelFusion/ComputeRoot 需评估; CrossLayerResidual 不可行（跨层残差依赖外部层输出）。

与 REQ-UMK-012 GroupMarker 交互: GroupMarker 标记层边界，层模板并行编译使用标记确定 group 边界。

§7.4 第二层：融合组级并行（中等收益, REQ-UMK-018）

核心洞察: FusionPlan 的 groups 形成生产-消费 DAG。同一拓扑层级的 groups 之间无数据依赖，可以并行 lower。

依赖分析器: GroupDependencyAnalyzer 分析 group 间依赖，返回拓扑层级列表：

  1. 对每个 group gi，收集其所有 op 的 input TensorId 集合
  2. 检查跨 group 的 output→input 依赖
  3. 拓扑排序：按依赖关系分层，每层的 groups 无互相依赖
  4. 循环依赖检测：无剩余 group 可分配时将剩余合并入同一层

约束: 层内 groups 有顺序依赖（attn → ffn）不能并行；emit_inter_group_casts 需要 producer group 先完成才能更新 resolver；组间 cast 在所有 groups 编译完成后统一处理；CrossLayerResidual 的 group 不能与其他组并行。

§7.5 第三层：自调优并行（精确优化, REQ-UMK-019）

核心洞察: 每个 GEMM 算子的 FMA 策略选择可以并行评估多种候选方案。

成本模型: ResourceBudget（l1_bytes/l2_bytes/avail_regs）+ FmaCandidate（strategy + estimated_cost）。成本评估维度：计算吞吐（FMA 单元利用率）、寄存器压力（spill penalty）、Cache 友好性（分块是否 fit L1/L2）。

并行评估: select_fma_best 收集 select_fma_candidates 的候选列表，使用 rayon into_par_iter 并行 evaluate_cost，选择 estimated_cmp 最优的策略。rayon 不可用时回退到串行评估。

§7.6 实施顺序

  1. 基础设施（已完成）: LoweringContext + 函数签名重构
  2. 第三层（已完成）: select_fma_best + ResourceBudget + 成本模型
  3. EmitState 抽取: 从 emit_fusion_groups 提取可变状态 → EmitState 结构体
  4. compile_layer_type_body 扩展: 支持 EpilogueInjection / TileLevelFusion / ComputeRoot
  5. 第一层（层模板并行）: LayerTemplate + append_with_mapping + rayon 并行编译
  6. 第二层（融合组并行）: GroupDependencyAnalyzer + 拓扑序并行 lower
