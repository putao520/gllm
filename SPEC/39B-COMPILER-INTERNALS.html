内部设计（内部实现，读者=gllm-kernels 开发者）

> 核心定位: 本文档定义 gllm-kernels 编译器的内部实现细节，包括 GroupMarker 机制、ForwardPhaseDispatch 三路分支、LoweringContext 数据结构、EmitState 可变状态、并行化策略、融合决策和 ISA Lowering。gllm-kernels 开发者需要遵守这些设计约束来维护编译器。

> 与 39A 的关系: 39A-COMPILER-CONTRACT 定义编译器外部接口契约（compile 入口、输入输出类型、禁止规则），读者是 gllm 开发者。

> 拆分说明: 原 SPEC 39-UNIFIED-MEGA-KERNEL 混合了外部接口与内部实现细节，本次修复按受众拆分为 39A（接口）和 39B（内部）。

完整模型编译时，CompileOutput 额外提供 MegaKernelWeightLayout 作为 pack 阶段的约定格式：

§1.3.2 GroupMarker 触发机制 (REQ-UMK-012)

编译器需要知道"哪些组属于同一层循环"来插入 LayerLoopBegin/LayerLoopEnd。这不是硬编码——而是从图的层结构推导：

/// Phase 2 融合引擎在组序列中插入结构标记。
/// 标记位置由 CompilerGraph 的层拓扑推导，不由编译器假设。
pub enum GroupMarker {
    /// 层循环开始。num_iterations 从图拓扑推导。
    LayerLoopBegin { num_iterations: usize },
    /// 层循环结束。
    LayerLoopEnd,
    /// ForwardPhaseDispatch 三路分支（仅多步生成图）。
    ForwardPhaseDispatch,
    /// 无标记——普通融合组。
    None,
}
推导规则：
  1. 图中连续 N 个同构子结构（相同 op 模式重复） → LayerLoopBegin { num_iterations: N }
  2. 单次出现的子结构 → 无循环标记，直接发射
  3. 图无 Argmax/StoreToken/CheckStopCondition → 无 ForwardPhaseDispatch
  4. 图有 Argmax → ForwardPhaseDispatch 插入在首次层循环前
不同图类型的标记序列：
图拓扑特征Group 序列标记
含 Argmax + 35 个同构子结构 + MHAembed + N*[attn+ffn] + lm_head + argmax + store + stop_checkForwardPhaseDispatch + LayerLoopBegin{35} + LayerLoopEnd
无 Argmax + 12 个同构子结构 + MeanPool/Classifyembed + 12*[attn+ffn] + pool/classifyLayerLoopBegin{12} + LayerLoopEnd
无 Argmax + 24 个同构子结构 + Classify/Rerankembed + 24*[attn+ffn] + classify + scoreLayerLoopBegin{24} + LayerLoopEnd
SG K-投影图rmsnorm + gemm无标记
SG Gather 图gather无标记
含 Argmax + 异构层（sliding/full 交替）+ AltUpembed + N*[altup+attn+ffn_sliding] + M*[altup+attn+ffn_global] + lm_head + ...ForwardPhaseDispatch + LayerLoopBegin{N} + LayerLoopEnd + LayerLoopBegin{M} + LayerLoopEnd
含 Argmax + MoE 子结构embed + N*[attn+moe_router+moe_ffn] + lm_head + argmax + store + stop_checkForwardPhaseDispatch + LayerLoopBegin{N} + LayerLoopEnd
含 Argmax + MLA + MoE + MTPembed + N*[mla_attn+moe_router+moe_ffn+shared_experts] + lm_head + mtp + argmax + stop_checkForwardPhaseDispatch + LayerLoopBegin{N} + LayerLoopEnd + MTP 融合组
验收标准

  2. Phase 2 融合引擎根据图同构子结构插入标记
  3. 不同图类型（encoder/decoder/SG 小图/MoE）标记序列正确
§1.3.3 ForwardPhaseDispatch 详细行为 (REQ-UMK-013) ForwardPhaseDispatch 是 JIT 内的三路分支，由 phase_id 寄存器控制： 
JIT 内部:
  mov phase_id, [rbp + phase_offset]     ; ABI 参数传入
  cmp phase_id, 0
  je prefill_path
  cmp phase_id, 1
  je decode_path
  ; else: single_pass_path (无生成循环)

prefill_path:
  <emit prefill groups>
  jmp after_layer_loop

decode_path:
  <emit decode groups + sampling + tokenstore + stop_check>
  jmp loop_back_or_exit

single_pass_path:
  <emit groups (pool/classify/rerank 等)>
  jmp epilogue
关键：三路分支的代码存在性由图拓扑决定：
  • 图无 Argmax → 无 decode_path 分支
  • 图无 Sampling ops → 无 decode_path
  • 图只有小图 ops → 无 ForwardPhaseDispatch，线性执行
验收标准
  1. CompileConfig 结构体定义中无 is_encoder/inference_mode/vocab_size/num_layers/hidden 字段
  2. vocab_size 从 OpKind::Argmax { vocab_size } 推导，不作为 CompileConfig 字段
  3. num_layers 从图拓扑推导，不作为 CompileConfig 字段
  4. num_eos_tokens 从 OpKind::CheckStopCondition 推导，不作为 CompileConfig 字段
  5. 无 lm_head_offset/lm_head_bytes 字段——权重偏移通过 named_offsets 映射表统一寻址
  6. 结构体命名为 CompileConfig（非 MegaKernelConfig）
  7. grep -rn "MegaKernelConfig" src/ 无结果
  8. CompileConfig 结构体中无 business_config 字段——BusinessConfig 已彻底分离，仅 BUILD 阶段使用
  9. debug_jit 为 CompileConfig 顶层字段（从 BusinessConfig 提升），编译器唯一直接读取的非图推导参数
ForwardPhaseDispatch 仅当图含 Argmax 算子时存在
图无 Argmax/StoreToken/CheckStopCondition → 无 ForwardPhaseDispatch 分支
无 Argmax 的图（含 MeanPool/Classify 但无 Argmax）无 ForwardPhaseDispatch
REQ-UMK-015 LoweringContext 编译会话共享状态

REQ-UMK-016 EmitState 可变状态抽取

REQ-UMK-017 层模板并行编译（第一层）

异构模型（Gemma-4）的多种层类型各自有独立的 fusion group 序列，层类型之间编译完全独立，可以用 rayon 并行编译。核心数据结构 LayerTemplate 封装编译后的 VmProgram + ABI 映射，compile_layer_type_body 编译单个层类型，VmProgram::append_with_mapping 将模板合并到主程序。并行化将编译时间从 O(总 groups) 降低到 O(最慢层类型)。与 REQ-UMK-012 GroupMarker 交互：GroupMarker 标记层边界，层模板并行编译使用标记确定 group 边界。

验收标准：
  1. LayerTemplate 结构体定义（body: VmProgram + abi_map: LayerAbiMap）
  2. LayerAbiMap 包含 input_ptr/weight_ptr/output_ptr/scratch_base 4 个 VRegId 映射
  3. compile_layer_type_body 函数签名：ctx + group_range + plan + graph + alloc + state + resolver → Result<LayerTemplate>
  4. VmProgram::append_with_mapping 方法实现：VReg 重映射 + ABI 指针替换
  5. rayon 并行编译路径：layer_types.par_iter().map(compile_layer_type_body).collect()
  6. 模板按顺序实例化：prog.append_with_mapping(ss_template, &ss_abi_subst) 等
  7. 同构模型（单层类型）走单路径编译，异构模型（Gemma-4）走并行模板编译
  8. cargo test --lib 无 LayerTemplate/append_with_mapping 相关测试失败

REQ-UMK-018 融合组级并行（第二层）

FusionPlan 的 groups 形成生产-消费 DAG，同一拓扑层级的 groups 之间无数据依赖，可以并行 lower。GroupDependencyAnalyzer 分析 group 间依赖，返回拓扑层级列表。依赖分析算法：收集每个 group 的 input TensorId，检查跨 group 的 output→input 依赖，拓扑排序按依赖关系分层。约束：层内 groups 有顺序依赖不能并行，组间 cast 需要所有 groups 编译完成后统一处理。

验收标准：
  1. GroupDependencyAnalyzer 结构体定义，analyze(plan, graph) → Vec<Vec<usize>> 拓扑层级列表
  2. 依赖分析算法：收集每个 group 的 input TensorId，检查跨 group 的 output→input 依赖
  3. 拓扑排序分层：同一层级的 groups 无互相依赖，可并行 lower
  4. 循环依赖检测：无剩余 group 可分配时将剩余合并入同一层
  5. lower_parallel_groups 函数签名：ctx + groups + plan + graph + alloc + resolver → Vec<(usize, VmProgram)>
  6. 层内 groups 有顺序依赖（attn → ffn）不能并行
  7. emit_inter_group_casts 在所有 groups 编译完成后统一处理
  8. cargo test --lib 无 GroupDependencyAnalyzer 相关测试失败

REQ-UMK-019 自调优并行（第三层）

每个 GEMM 算子的 FMA 策略选择可以并行评估多种候选方案。select_fma_best 收集 select_fma_candidates 的候选列表，使用 rayon into_par_iter 并行 evaluate_cost，选择 estimated_cmp 最优的策略。当 rayon 不可用时回退到串行评估。这是三层并行化中实现优先级最低（第三层）但已完成的层级。

验收标准：
  1. ResourceBudget 结构体定义：l1_bytes/l2_bytes/avail_regs 3 字段
  2. FmaCandidate 结构体定义：strategy + estimated_cost
  3. select_fma_best 使用 rayon into_par_iter 并行 evaluate_cost
  4. select_fma_candidates 返回候选列表，evaluate_cost 计算 estimated_cmp
  5. rayon 不可用时回退到串行评估（into_iter 替代 into_par_iter）
  6. 成本评估维度：计算吞吐 + 寄存器压力 + Cache 友好性
  7. 此层级已在代码中实现（isa_hook.rs 中 select_fma_best）

§X GPU Dialect Lowering（GPU ISA 间接分层）

定位：CPU 和 GPU 在 ISA Lowering 阶段架构不同。CPU 直接 emit 原生机器码，GPU 通过 trait 间接层生成 ISA 文本。

架构对比

路径流程产出
CPU 直接 LoweringVmInstr → x86_lower / aarch64_lower → mmap 机器码Raw machine code bytes
GPU Dialect LoweringVmInstr → GpuLower → GpuDialect trait → ISA 文本 → 驱动编译Compiled GPU kernel

Entity: GpuDialect

Trait 接口定义 GPU vendor 语义差异。三个独立 impl 覆盖主流 GPU 厂商。

字段类型必填约束说明
emit_loop_begintrait method是发射循环开始指令（PTX: @%p label / HIP: s_branch / Metal: label:）
emit_loop_endtrait method是发射循环结束指令
emit_barriertrait method是发射屏障（PTX: bar.sync / HIP: s_barrier / Metal: threadgroup_barrier）
emit_shared_mem_alloctrait method是分配共享内存（.shared / .lds / threadgroup）
emit_shared_mem_storetrait method是共享内存写入
emit_shared_mem_loadtrait method是共享内存读取
emit_warp_reducetrait method是warp 级规约（PTX: shfl.sync / HIP: ds_wrhs / Metal: simd_reduce）
emit_tmadescriptor_inittrait method否SM 9.0+TMA 描述符初始化（仅 PTX/Hopper+）
emit_tma2d_copytrait method否SM 9.0+TMA 2D 拷贝（仅 PTX/Hopper+）
emit_tile_mmatrait method否SM 8.0+Tensor Core tile MMA（PTX: mma.async / HIP: mfma / Metal: simd_matrix）

三个 impl 实现

Impl目标硬件SM 版本ISA 文本格式
PtxDialectNVIDIA GPUSM 6.1 - 12.0NVIDIA PTX（汇编文本）
HipDialectAMD GPURDNA 1-4 / CDNAAMD GCN ISA（汇编文本）
MetalDialectApple GPUApple GPU family 1-9Metal AIR（中间表示）

PtxKernelRegistry：按 SM 版本选择最优 PTX kernel。禁止 Fallback（CLAUDE.md GPU GEMM 全 JIT 铁律）。SM 6.1 / 8.0 / 9.0 / 10.0 / 12.0 各有独立 PTX kernel 集。

设计原则：CPU 直出机器码（性能最优，无中间文本开销），GPU 通过 ISA 文本+驱动编译（GPU 硬件要求 ISA 通过驱动加载，无法直接 mmap）。两种路径共享 VmInstr IR，差异仅在 Phase 3 ISA Lowering。
