策略仲裁器 (Strategy Arbiter)

> SSOT 声明: 本文档是 gllm 推理引擎全局优化策略优先级决策的唯一真源。定义 SchedulingMode（延迟/吞吐模式）、GraphProfile（模型图拓扑特征提取）、StrategyArbiter（策略优先级仲裁），及其与 HwOptEngine（02-HARDWARE.md §10）的集成关系。

1. 动机

1.1 问题

当前 HwOptEngine 的策略决策完全由硬件参数驱动（DeviceProfile + MemoryNetworkSensors + ProbeResult）。这导致两个盲区：

1. 使用场景盲区: 同一硬件上，本地单请求低延迟推理与服务端高吞吐批处理的最优策略完全相反，但 HwOptEngine 无法区分。

2. 模型图拓扑盲区: Dense 大模型（Llama-70B）与 MoE 稀疏模型（DeepSeek-V3）的最优资源分配策略截然不同，但 HwOptEngine 仅通过 ModelConfig 获取标量参数（hidden_dim、num_layers），未分析图级拓扑特征。

1.2 解决方案

引入 Strategy Arbiter 作为 HwOptEngine 的前置阶段。Arbiter 输出 StrategyBias——一组归一化的资源配额权重，注入 HwOptEngine 的 CostModel，使所有下游求解器的成本评估函数自动偏向场景最优策略。

1.3 决策链

```

SchedulingMode (用户指定)

└→ GraphProfiler (模型图静态分析)

└→ StrategyArbiter (Mode × GraphProfile × DeviceProfile → StrategyBias)

└→ HwOptEngine (StrategyBias 注入 CostModel → HwOptPlan)

└→ JIT Codegen (生成机器码)

```

```rust
pub fusion_profitable: f64,

/// 流水线价值度 (高 = 小 GEMM, memory-bound, 深层数)

pub pipeline_valuable: f64,

}

```

REQ-ARB-002: GraphArchetype 的五个维度之和不要求为 1.0。每个维度独立归一化，表示该维度的绝对强度。

4. StrategyArbiter — 策略仲裁器

4.1 职责

接收 SchedulingMode × GraphArchetype × DeviceProfile，输出 StrategyBias——一组成本模型调制系数，注入 HwOptEngine 使其自动偏向场景最优策略。

核心理念: Arbiter 不直接决定具体策略（那是 HwOptEngine 各 Solver 的职责），而是通过调整成本函数的权重间接影响所有决策。这保持了 HwOptEngine 的 Cost-Model 驱动架构不变。

4.2 StrategyBias 定义

```rust

/// 策略偏置系数。注入 HwOptEngine CostModel，调制成本评估。

/// 所有系数 > 0.0，1.0 = 中性，< 1.0 = 偏好（降低该策略成本），> 1.0 = 惩罚（提高成本）。

pub struct StrategyBias {

// ── 资源配额偏置 ──

/// 融合族偏置: < 1.0 时 CostModel 低估融合的代价（偏好融合）

pub fusion_cost_scale: f64,

/// 流水线族偏置: < 1.0 时 CostModel 低估 pipeline 的代价（偏好双缓冲/预取）

pub pipeline_cost_scale: f64,

/// 并行族偏置: < 1.0 时 CostModel 低估 wave 分裂的代价（偏好多 wave）

pub parallelism_cost_scale: f64,

// ── 寄存器预算偏置 ──

/// epilogue 深度偏好: > 1.0 时 FusionSolver 更倾向深 epilogue（牺牲 k_depth）

pub epilogue_depth_preference: f64,

/// k_depth 偏好: > 1.0 时 GemmSolver 更倾向深 k pipeline（牺牲 epilogue）

pub k_depth_preference: f64,

// ── 缓存预算偏置 ──

/// KV cache 预算比例调制 (> 1.0 = 给 KV 更多 L2)

pub kv_cache_budget_scale: f64,

/// 权重预取预算比例调制

pub weight_prefetch_budget_scale: f64,

// ── 批处理偏置 ──

/// batch 大小偏好: 0.0 = 永远 batch=1, 1.0 = BatchSolver 自由决策

pub batch_flexibility: f64,

/// decode/prefill 混合比偏置

pub decode_ratio_scale: f64,

// ── MoE 专属偏置 ──

/// 专家驻留激进度: 0.0 = 全部常驻, 1.0 = 激进驱逐

pub expert_eviction_aggressiveness: f64,

/// 专家权重预取优先级: > 1.0 = 更积极预取即将使用的专家

pub expert_prefetch_priority: f64,

// ── 特殊策略偏置 ──

/// 投机解码价值: > 1.0 = 更倾向启用 speculative decoding

pub speculative_decoding_value: f64,

/// 量化激进度: > 1.0 = 更倾向低比特量化

pub quantization_aggressiveness: f64,

}

```

4.3.2 GraphArchetype 调制

在基线偏置上，根据模型图原型做乘法调制：

```

// 融合族调制

bias.fusion_cost_scale *= lerp(1.0, 0.6, archetype.fusion_profitable)

// fusion_profitable 高 → 融合代价进一步降低 → 更偏好融合

// 流水线族调制

bias.pipeline_cost_scale *= lerp(1.0, 0.6, archetype.pipeline_valuable)

// pipeline_valuable 高 → pipeline 代价进一步降低

// 并行族调制

bias.parallelism_cost_scale *= lerp(1.0, 0.5, archetype.parallelism_exploitable)

// parallelism_exploitable 高 → 并行代价进一步降低

// 寄存器竞争调制 (融合 vs 流水线的寄存器分配)

reg_tension = archetype.fusion_profitable - archetype.pipeline_valuable

if reg_tension > 0.0:

// 融合更有价值 → 寄存器给 epilogue

bias.epilogue_depth_preference *= 1.0 + reg_tension × 0.5

bias.k_depth_preference *= 1.0 - reg_tension × 0.3

else:

// 流水线更有价值 → 寄存器给 k_depth

bias.k_depth_preference *= 1.0 + (-reg_tension) × 0.5

bias.epilogue_depth_preference *= 1.0 - (-reg_tension) × 0.3

// MoE 调制

if archetype.parallelism_exploitable > 0.5: // MoE 模型

bias.expert_eviction_aggressiveness *= lerp(1.0, 1.5, archetype.memory_intensive)

bias.expert_prefetch_priority *= lerp(1.0, 2.0, archetype.memory_intensive)

// 内存密集的 MoE → 更激进驱逐 + 更激进预取

// KV cache 调制

bias.kv_cache_budget_scale *= lerp(1.0, 1.5, archetype.memory_intensive)

// 量化调制

bias.quantization_aggressiveness *= lerp(1.0, 1.3, archetype.memory_intensive)

// 内存密集 → 更激进量化

```

```

// GPU 有海量寄存器 → 寄存器竞争缓解
```

12. Throughput 模式 effective_M 循环依赖解决

12.1 问题

RooflineAnalyzer（Level 0）需要 effective_M 计算算术强度，但精确的 batch size 由 BatchSolver（Level 3）决定。这形成循环依赖。

12.2 解决方案: 两阶段求解

```

阶段 1: 使用估算 M 完成初始求解

estimated_M = estimate_batch_size(mode, profile, model)

roofline = RooflineAnalyzer::analyze(profile, model, estimated_M)

→ 完成 Level 0~4 全部求解 → 初始 HwOptPlan

阶段 2: 用初始 BatchPlan 的实际 batch size 校正（仅当偏差显著时）

actual_M = initial_plan.batch.decode_slots

if abs(actual_M - estimated_M) / estimated_M > 0.5: // 偏差超过 50%

重新用 actual_M 跑 RooflineAnalyzer → 重新求解（最多 1 轮校正）

```

12.3 estimate_batch_size() 估算函数

```rust

fn estimate_batch_size(mode: SchedulingMode, profile: &DeviceProfile, model: &ModelConfig) -> usize {

match mode {

SchedulingMode::Latency => 1, // 永远是 1，无需估算

SchedulingMode::Throughput => {

// 目标: GPU 内存的 60% 给 KV cache + 权重，剩余给 batch

// batch_M ≈ remaining_memory / (kv_per_token × max_seq_len)

// 保守估算: min(64, gpu_memory_gb × 4)

// CPU: min(32, num_cores × 2)

if profile.is_gpu() {

let gpu_mem_gb = profile.total_memory_bytes() / (1 << 30);

(gpu_mem_gb * 4).min(256).max(8) as usize

} else {

let cores = profile.num_cores();

(cores * 2).min(64).max(4)

}

}

}

}

```

REQ-ARB-011: Throughput 模式使用两阶段求解打破循环依赖。Latency 模式 effective_M=1，无循环。校正最多执行 1 轮，禁止迭代收敛。

14. CostModel 注入方向规则

核心原则: StrategyBias 的 *_cost_scale 系列字段遵循统一语义——乘到代价侧，除到收益侧。

```

代价侧 (越低越好):

adjusted_cost = base_cost × bias.*_cost_scale

// cost_scale < 1.0 → 代价降低 → Solver 更倾向选择该策略

收益侧 (越高越好):

adjusted_benefit = base_benefit × (1.0 / bias.*_cost_scale)

// cost_scale < 1.0 → 1/scale > 1.0 → 收益放大 → Solver 更倾向选择该策略

```

方向一致性: 对于同一个 bias 字段，代价侧和收益侧的调制方向必须一致。fusion_cost_scale = 0.5 意味着"融合的代价减半"和"融合的收益翻倍"，二者效果相同。

16. 常见实现错误与反模式

16.1 ❌ 注入方向搞反

```rust

// ❌ 错误: fusion_cost_scale < 1.0 时反而提高了融合代价

effective_savings = fusion_savings × bias.fusion_cost_scale; // 应该是 ÷

// ✅ 正确: fusion_cost_scale < 1.0 时融合收益放大

effective_savings = fusion_savings × (1.0 / bias.fusion_cost_scale);

// 或等价地: effective_savings = fusion_savings / bias.fusion_cost_scale;

```

16.2 ❌ 直接读 SchedulingMode 做分支

```rust

// ❌ 错误: 绕过 StrategyBias

if plan.inference_mode == SchedulingMode::Latency {

self.max_batch = 1;

self.disable_continuous_batching();

}

// ✅ 正确: 通过 bias 传导

self.max_batch = (base_max as f64 * plan.strategy_bias.batch_flexibility) as usize;

// Latency mode: batch_flexibility = 0.0 → max_batch = 0 → 实际 max(1, 0) = 1

```

注意: batch_flexibility = 0.0 会导致 max_batch = 0，调用方必须 max(1, computed)。

16.3 ❌ validate() 改为 panic

```rust

// ❌ 错误: 数值计算边界值不应 panic

fn validate(&self) -> Result<(), ArbiterError> {

if self.fusion_cost_scale <= 0.0 { return Err(...); }

}

// ✅ 正确: clamp 到合法范围

fn validate(&mut self) {

self.fusion_cost_scale = self.fusion_cost_scale.clamp(0.2, 3.0);

}

```

16.4 ❌ sigmoid 用错形式

```rust

// ❌ 错误: 用 tanh 代替 sigmoid

fn sigmoid(x: f64) -> f64 { x.tanh() } // 值域 (-1, 1)，不是 (0, 1)

// ❌ 错误: 用 ReLU 代替 sigmoid

fn sigmoid(x: f64) -> f64 { x.max(0.0).min(1.0) } // 线性 clamp，不是 S 曲线

// ✅ 正确: 标准逻辑函数

fn sigmoid(x: f64) -> f64 { 1.0 / (1.0 + (-x).exp()) }

```

16.5 ❌ GraphArchetype 维度相加归一化

```rust

archetype.compute_intensive = c / total;

// 这会丢失绝对强度信息！五维独立，不要求和为 1.0

// ✅ 正确: 每个维度独立 clamp 到 [0, 1]

archetype.compute_intensive = clamp(compute_raw, 0.0, 1.0);
```

IntentBias 是用户意图偏好层，不暴露 StrategyBias 实现细节。用户通过 IntentBias 表达意图（如"我更关心延迟"），系统翻译为 StrategyBias 调制系数。IntentBias 包含 5 个字段：scenario(ScenarioHint)、comm_overlap(OverlapHint)、decode_sm_ratio(Option<f32>)、kv_budget_scale(Option<f32>)、quant_aggression(Option<f32>)。所有字段有 Default（scenario=Auto, comm_overlap=Auto, 三个 Option=None）。Option 字段=None 时不覆盖自动推导值，Some 时直接覆盖。设计遵循平权交付原则，无主次分期。

IntentBias 结构体包含 5 个字段且所有字段有 Default
Option 字段=None 时不覆盖自动推导值
Option 字段=Some 时直接覆盖 StrategyBias 对应字段
不暴露 StrategyBias 的 fusion_cost_scale/batch_flexibility/expert_eviction_aggressiveness/expert_prefetch_priority 等内部字段
Default::default() 提供零配置体验，不设 .intent() 时行为与现有完全一致
验收标准：
  1. IntentBias 结构体包含 5 个字段（scenario/comm_overlap/decode_sm_ratio/kv_budget_scale/quant_aggression），所有字段有 Default 实现
  2. Option 字段=None 时 StrategyBiasResolver 不覆盖 StrategyArbiter 自动推导值，行为与不设 IntentBias 完全一致
  3. Option 字段=Some(value) 时 StrategyBiasResolver 直接覆盖 StrategyBias 对应字段，值从 IntentBias 传递到 StrategyBias 无丢失
  4. IntentBias 不暴露 StrategyBias 的内部字段（fusion_cost_scale/batch_flexibility/expert_eviction_aggressiveness/expert_prefetch_priority），用户可见字段仅为 5 个设计字段
  5. Default::default() 返回 scenario=Auto, comm_overlap=Auto, decode_sm_ratio=None, kv_budget_scale=None, quant_aggression=None，零配置体验：不设 .intent() 时行为与现有完全一致

REQ-IB-002 ScenarioHint 场景偏好枚举

ScenarioHint 是 6 值枚举，覆盖 90% 场景偏好。枚举值：Auto(默认，系统自动推导)、LatencyCritical(极致延迟，等效 batch_size=1 基线)、ThroughputOptimal(极致吞吐，等效 batch_size>1 基线)、LongContext(长上下文，KV cache 优先+低量化激进度)、DistributedHeavy(分布式密集，通信重叠优先+专家预取激进)、MemoryConstrained(内存受限，激进量化+专家驱逐+KV 压缩)。ScenarioHint 映射到 StrategyArbiter 的 InferenceMode 基线，不是简单替换 InferenceMode，而是重新走 mode_baseline + archetype_modulation + hardware_adjustment，只是用 scenario 对应的基线替代 batch_size 推导的基线。这保持了 GraphArchetype 和 DeviceProfile 的调制效果。

ScenarioHint 枚举包含 6 个值：Auto/LatencyCritical/ThroughputOptimal/LongContext/DistributedHeavy/MemoryConstrained
Auto 为 Default，保持现有 batch_size 推导链
LatencyCritical 强制 InferenceMode::Latency 基线
ThroughputOptimal 强制 InferenceMode::Throughput 基线
LongContext → Latency 基线 + kv_cache_budget_scale *= 1.5 + quantization_aggressiveness *= 0.7
DistributedHeavy → Throughput 基线 + expert_prefetch_priority *= 2.0 + pipeline_cost_scale *= 0.6
MemoryConstrained → Throughput 基线 + quantization_aggressiveness *= 1.5 + expert_eviction_aggressiveness *= 1.5 + kv_cache_budget_scale *= 0.7
scenario 不是简单替换 InferenceMode，而是重新走完整仲裁管线
验收标准：
  1. ScenarioHint 枚举包含 6 个值：Auto/LatencyCritical/ThroughputOptimal/LongContext/DistributedHeavy/MemoryConstrained
  2. Auto 为 Default，保持现有 batch_size 推导链，StrategyArbiter 输出与不设 scenario 完全一致
  3. LatencyCritical 强制 InferenceMode::Latency 基线，batch_size 推导被覆盖
  4. ThroughputOptimal 强制 InferenceMode::Throughput 基线，batch_size 推导被覆盖
  5. LongContext → Latency 基线 + kv_cache_budget_scale *= 1.5 + quantization_aggressiveness *= 0.7
  6. DistributedHeavy → Throughput 基线 + expert_prefetch_priority *= 2.0 + pipeline_cost_scale *= 0.6
  7. MemoryConstrained → Throughput 基线 + quantization_aggressiveness *= 1.5 + expert_eviction_aggressiveness *= 1.5 + kv_cache_budget_scale *= 0.7
  8. scenario 重新走完整仲裁管线（mode_baseline + archetype_modulation + hardware_adjustment），不是简单替换 InferenceMode

REQ-IB-004 StrategyBiasResolver 三阶段管线

StrategyBiasResolver 是三阶段管线，将自动推导的 StrategyBias 与用户 IntentBias 合并。阶段 1：auto_bias = StrategyArbiter::arbitrate(batch_size, archetype, hw) 获取自动推导基线。阶段 2：intent 覆盖——scenario 覆盖 InferenceMode 基线（重新 mode_baseline + archetype + hw）；comm_overlap 覆盖 MkVariant 选择 + CommScheduleHint；decode_sm_ratio 直接覆盖 SmPartitionConfig.decode_ctas/total_ctas 比例；kv_budget_scale 直接覆盖 StrategyBias.kv_cache_budget_scale；quant_aggression 直接覆盖 StrategyBias.quantization_aggressiveness。阶段 3：bias.validate()——clamp 所有字段到合法范围。关键：scenario 不是简单替换 InferenceMode，而是重新走 mode_baseline + archetype_modulation + hardware_adjustment，只是用 scenario 对应的基线替代 batch_size 推导的基线。内部接口：API-STRATEGY-RESOLVE-BIAS (POST /internal/strategy/resolve_bias)。

StrategyBiasResolver 实现 resolve(intent_bias, batch_size, archetype, hw) → StrategyBias
阶段 1：调用 StrategyArbiter::arbitrate() 获取 auto_bias
阶段 2：IntentBias 各字段按规则覆盖 auto_bias
scenario 覆盖：重新走完整仲裁管线，使用 scenario 对应基线替代 batch_size 推导基线
comm_overlap 覆盖：影响 MkVariant 选择和 CommScheduleHint
decode_sm_ratio 覆盖：直接设置 SmPartitionConfig.decode_ctas/total_ctas 比例
kv_budget_scale 覆盖：直接设置 StrategyBias.kv_cache_budget_scale
quant_aggression 覆盖：直接设置 StrategyBias.quantization_aggressiveness
阶段 3：bias.validate() clamp 所有字段到合法范围
decode_sm_ratio 覆盖后必须 is_valid() 检查 SmPartitionConfig 有效性
验收标准：
  1. OverlapHint 枚举包含 5 个值：Auto/PreferOverlap/PreferIsolated/ForceDoubleBuffer/ForceFlux
  2. Auto 为 Default，使用 select_mk_variant(sm_version, total_sm) 现有逻辑，MkVariant 选择与不设 comm_overlap 完全一致
  3. PreferOverlap → SM90+ 选 Cluster5x3(prefill 更重)，SM70-89 选 GridSync，SM60-69 选 GridSync，SM<60 选 Serial
  4. PreferIsolated → SM90+ 选 Cluster6x2(decode 更重)，SM70-89 选 GridSync(decode_ratio 调制实现隔离语义)，SM60-69 选 GridSync，SM<60 选 Serial
  5. ForceDoubleBuffer → CommScheduleHint.overlap_mode = DoubleBuffer，不影响 MkVariant 选择
  6. ForceFlux → CommScheduleHint.overlap_mode = FluxOverDecomposition，不影响 MkVariant 选择
  7. ForceFlux 在单卡场景降级为 Auto + warning log（输出目标：tracing::warn；格式：gllm::intent: ForceFlux ignored on single-GPU, falling back to Auto；不影响编译结果，仅运行时提示）

REQ-IB-007 ParallelConfig 并行策略配置

ParallelConfig 定义模型切分加载策略。字段：tp_size(u32, Tensor Parallelism 度，默认 1)、pp_size(u32, Pipeline Parallelism 度，默认 1)、ep_size(u32, Expert Parallelism 度，默认 1)、rank(u32, 本节点 rank，默认 0)、world_size(u32, 总 rank 数，默认 1)、unique_id(UniqueId, 通信组标识，由 get_unique_id() 生成)。约束：tp_size * pp_size * ep_size == world_size（否则 build() 返回 Err）。tp_size 影响 CompilerGraph 构建——只包含 rank 对应的 attention head 子集。pp_size 影响 CompilerGraph 构建——只包含 rank 对应的 layer 范围。ep_size 影响 MoE 权重加载——只加载 rank 对应的 expert 子集。unique_id 必须在所有 rank 间共享（一个 rank 调用 get_unique_id()，结果分发到其他 rank）。

pp_size>1 时 CompilerGraph 仅包含 rank 对应的 layer 范围

unique_id 必须在所有 rank 间共享

REQ-IB-008 PdDisaggConfig Prefill-Decode 分离配置

PdDisaggConfig 定义 Prefill-Decode 分离部署策略。字段：mode(PdDisaggMode, 默认 Collocated)、role(NodeRole, 默认 Auto)。PdDisaggMode 枚举：Collocated(PD 同机，当前默认行为)、Disaggregated(PD 分离部署，prefill 节点和 decode 节点物理分离)。NodeRole 枚举：Auto(自动，Collocated 下忽略)、PrefillOnly(只做 prefill，MegaKernel 仅编译 prefill_path)、DecodeOnly(只做 decode，MegaKernel 仅编译 decode_path + KV 接收)、Mixed(Collocated 模式下的混合调度)。Disaggregated 模式下 KV Cache 必须通过 gllm-nccl 分布式分页传输（REQ-DP-001~014），prefill 节点产出 KV 后通过 P2pPageFetch/RdmaPageFetch 传给 decode 节点。Disaggregated 模式要求 nccl feature 启用。

REQ-IB-013 DistributedConfig 编译时策略映射

DistributedConfig 的所有配置编译时 bake 进机器码，运行时零分支。映射链路：1) ParallelConfig.tp_size → CompilerGraph 按 head 分片构建 → 不同 rank 产出不同图 → compile() 产出不同的 MegaKernelFn。2) ParallelConfig.pp_size → CompilerGraph 按 layer 范围分片 → 层间通过 AllReduceChunk/CommBarrier 同步。3) ParallelConfig.ep_size → MoE 权重按 expert_id 子集加载 → ExpertWeightPrefetcher 只管理本地 expert。4) PdDisaggConfig.mode=Disaggregated+PrefillOnly → compile() 只编译 prefill_path → MegaKernelFn 不含 decode 循环和 Argmax。5) PdDisaggConfig.mode=Disaggregated+DecodeOnly → compile() 只编译 decode_path + KV 接收指令 → MegaKernelFn 从 CommHandle 接收 KV page。6) KvDistributionConfig.mode=Mirror → 编译时生成 PrefetchAllKvPages 批量 P2pPageFetch 指令。7) KvDistributionConfig.mode=PartialHeadMirror → 热点 head 走本地 VmInstr，冷 head 起远程 RemotePageAttn，编译时按 mirror_heads 参数分配。8) CommConfig.compress=AlwaysCompress → RdmaPageFetchCompressed 替代 RdmaPageFetch。9) MoeDistributedConfig.expert_placement=HotCold → 热专家权重常驻 HBM（编译时 allocate），冷专家走 RDMA PrefetchPipeline。关键原则：配置不同 → 图不同 → 机器码不同，同一配置 → 同一机器码（JIT 缓存命中）。
